# 11 — The Great Mapping Elimination and the MCP Session Migration to Redis

*March 22, 2026*

---

## Setting the Stage

Journal 10 ended with a question and a plan. Two variants sat on the table — 008.A (eliminate the mapping entirely) and 008.B (move the mapping to Redis) — and the codebase still had a `McpSessionMapping` class holding a `ConcurrentDictionary` that blocked any hope of running multiple FinWise instances behind a load balancer.

Today we stopped planning and started cutting.

---

## The Spec Review That Found Real Issues

Before touching any code, a deep review of the 008.A spec surfaced three problems that would have caused real implementation mistakes:

**The sliding TTL lie.** The spec's Redis key design table labeled `mcpinit:*` keys as "24h sliding" — but the proposed code never refreshed the TTL on read. `AllowSessionMigrationAsync` would read the init params from Redis but leave the expiration clock ticking from the original write. A session initialized at hour 0 could lose its migration data at hour 24, even if instances were actively migrating it. The fix: add `KeyExpireAsync` after every successful read.

**The `originalAgentSessionId` ghost.** Under 008.A, `agentSessionId` is never mutated — the whole point is keeping the same ID across resets. But the spec's Step 4 didn't mention removing the `originalAgentSessionId` variable that existed solely for the old "generate new GUID on reset" path. An implementer would leave confusing dead code in the catch blocks.

**The missing test rewrite.** `ResetSessionAsync_ReturnsNewSessionId` doesn't just need an "update" — the return type changes from `Task<string>` to `Task`. That's a full rewrite, not a tweak. The spec originally said "update existing tests" which understated the impact.

## Six Steps, One by One

The implementation followed the spec's six steps with ACK gates before each one — no step executed without explicit approval.

### Step 1: The SDK Upgrade

MCP SDK `0.5.0-preview.1` → `1.1.0` GA. Two lines changed in `Directory.Packages.props`. Build passed. All 78 tests passed. The breaking changes audit in the spec was correct — zero impact on FinWise.

The `ISessionMigrationHandler` interface was verified via reflection against the installed 1.1.0 assembly — method signatures matched the spec exactly:

```
OnSessionInitializedAsync(HttpContext, string, InitializeRequestParams, CancellationToken)
AllowSessionMigrationAsync(HttpContext, string, CancellationToken)
```

### Step 2: The Migration Handler

Creating `RedisSessionMigrationHandler` hit an immediate problem: the existing `Infrastructure.cs` file in the McpServer project defined a class called `Infrastructure` in the `FinWise.McpServer` namespace. Adding an `Infrastructure/` folder created a C# namespace collision — exactly the scenario the AGENTS.md rule "sub-folder names must NOT match class names" was written to prevent.

The fix was clean: rename `Infrastructure.cs` → `ServerSetup.cs` with the class becoming `ServerSetup`. Three call sites in `Program.cs` updated. The irony: this class would itself be decomposed later in the session.

A subtler issue: `JsonSerializer.Deserialize<InitializeRequestParams>(json!)` was ambiguous between `ReadOnlySpan<byte>` and `string` overloads. Redis returns a `RedisValue` whose implicit conversions confused the compiler. The fix: `.ToString()` to force the string overload — same pattern used in `RedisAgentSessionStore`.

### Step 3: Simplifying the Tools

> "I don't like the name `McpSessionHelper`, with 'Helper' — can you find a better domain-oriented name?"

The new name: `McpSessionAccessor` — following the ASP.NET pattern (`IHttpContextAccessor` *accesses* the HttpContext).

`FinWiseTools.cs` went from this:

```csharp
var sessionMapping = serviceProvider.GetRequiredService<McpSessionMapping>();
var sessionId = McpSessionMapping.GetSessionId(httpContext);
var agentSessionId = sessionMapping.GetOrCreateAgentSessionId(sessionId);
var result = await workflowService.ProcessMessageAsync(agentSessionId, query);
if (result.WasReset)
    sessionMapping.UpdateAgentSessionId(sessionId, result.AgentSessionId);
```

To this:

```csharp
var mcpSessionId = McpSessionAccessor.GetSessionId(httpContext);
var result = await workflowService.ProcessMessageAsync(mcpSessionId, query);
```

Five lines of mapping ceremony → two lines of direct pass-through.

### Step 4: Changing What "Reset" Means

The core semantic change of 008.A: a reset clears data under the same session ID instead of generating a new GUID.

```csharp
// Before: generates new identity
await _sessionManager.ClearSessionAsync(agentSessionId);
agentSessionId = Guid.NewGuid().ToString();

// After: clears data, keeps identity
await _sessionManager.ClearSessionAsync(agentSessionId);
```

The `originalAgentSessionId` variable — created to track the pre-mutation ID for catch blocks — became dead code. Both catch blocks simplified from `originalAgentSessionId` to `agentSessionId`.

`ResetSessionAsync` changed from `Task<string>` to `Task`. The existing test `ResetSessionAsync_ReturnsNewSessionId` became `ResetSessionAsync_ClearsSessionWithoutNewId` — verifying that the same ID still works after reset.

### Step 5: The Delete

`McpSessionMapping.cs` deleted. The DI registration in `Program.cs` replaced by `ISessionMigrationHandler` registration — conditional on Redis being enabled:

```csharp
if (redis is not null)
{
    // MCP session migration: stores MCP initialize handshake params in Redis (mcpinit:* keys)
    // so any instance can reconstruct the MCP transport session. Separate from agent session
    // storage (agentsession:* keys) which holds conversation state.
    builder.Services.AddSingleton<ISessionMigrationHandler>(
        new RedisSessionMigrationHandler(redis, TimeSpan.FromMinutes(redisOptions.SessionTtlMinutes)));
}
```

The comment was deliberate — separating *MCP session* Redis keys from *agent session* Redis keys in the reader's mind.

### Step 6: The Test Project

A new `FinWise.McpServer.UnitTests` project born with 8 tests for `RedisSessionMigrationHandler` — mocked Redis, same pattern as `RedisAgentSessionStoreTests`. Later, 3 tests for `McpSessionAccessor` (header present, missing, empty) brought it to 11.

## The Infrastructure Decomposition

With the spec implementation done, attention turned to the growing `Program.cs` — 170 lines of inline infrastructure creation that mixed Azure OpenAI, CosmosDB, Redis, Azure AI Foundry, and MCP concerns in one scroll.

> "Should we move/split the ServerSetup class and put what's related to infrastructure in segregated folders and classes into the /Infrastructure folder? Do it per domains/infra-area, do not mix concerns. Follow SRP."

Six factories emerged, each owning a single infrastructure concern:

```
src/FinWise.McpServer/Infrastructure/
├── AgentSessionStorage/
│   └── AgentSessionStoreFactory.cs
├── AzureAIFoundry/
│   └── StockAgentFactory.cs
├── AzureOpenAI/
│   └── AzureOpenAIChatClientFactory.cs
├── Logging/
│   └── LoggingSetup.cs
├── McpSession/
│   ├── McpSessionAccessor.cs
│   └── Redis/
│       └── RedisSessionMigrationHandler.cs
└── UserProfileStorage/
    └── UserProfileStoreFactory.cs
```

`Program.cs` shrank from 170 to 95 lines — pure composition root, no infrastructure details.

A Critic review caught two things: `HandleServerException` had been silently renamed from `HandleMcpServerException` (fixed), and `StockAgentFactory` lacked the diagnostic env-var logging that `AzureOpenAIChatClientFactory` had (added for consistency).

The naming evolved through dialogue:

> "SessionStoreFactory and SessionStorage should have the prefix like AgentSessionStore so there's no confusion with the MCP sessions."

`SessionStorage/` → `AgentSessionStorage/`, `ProfileStorage/` → `UserProfileStorage/`. Clear disambiguation from `McpSession/`.

> "Should the McpSessionAccessor.cs be in the root? Or in a folder named McpSession?"

Into `Infrastructure/McpSession/` — alongside the Redis migration handler, following the same pattern as `AgentSessionStores/` in the MultiAgentWorkflow project where the interface sits at the concern level and the provider in a subfolder.

## Making the Stock Agent Optional

An insight from the Critic review:

> "StockAgentFactory.CreateStockAgentAsync() failure blocks startup entirely — even when a user only wants profile/advisor functionality."

The fix was minimal — 4 files, ~15 lines:

- `CreateStockAgentAsync()` → `TryCreateStockAgentAsync()` — returns `null` with a warning instead of throwing
- `FinWiseWorkflowService` accepts `AIAgent?` — conditionally includes stock agent in the workflow
- The server starts without Azure AI Foundry credentials, running profile + advisor agents only

| Test | What It Proves |
|------|---------------|
| Roundtrip | Store on init, restore on migration — the core scale-out contract |

All 12 Redis integration tests pass (6 existing + 6 new). Combined with 8 McpServer integration tests, 11 McpServer unit tests, and 78 MultiAgentWorkflow unit tests: **109 tests, all green**.

## What Got Deleted

| File/Class | Why It Existed | Why It's Gone |
|------------|---------------|---------------|
| `McpSessionMapping.cs` | Maps MCP session → agent session ID | MCP Session ID *is* the agent session ID now |
| `ServerSetup.cs` | Monolithic static class for Azure OpenAI + logging | Split into 6 SRP factory classes |
| `Infrastructure.cs` | Original name before namespace collision | Renamed to `ServerSetup.cs`, then decomposed |
| `Guid.NewGuid()` in reset | Generated new identity on every reset | Reset clears data under the same key |
| `originalAgentSessionId` | Tracked pre-mutation ID for catch blocks | `agentSessionId` never mutates — dead variable |

## Reflection

The mapping was never necessary. Looking back at journal 02 where `McpSessionMapping` was created, it existed only because `ProcessMessageAsync` was written as "generate new GUID on reset" rather than "clear data, keep same key." That early design choice created a problem, then introduced a class to solve it.

008.A didn't just solve the scale-out problem — it corrected an accidental complexity that was there from the beginning. The simplest code is the code that was never needed.

The infrastructure refactoring was a bonus: `Program.cs` as a pure composition root — calling factories, wiring DI, nothing else — is the cleanest version of this file the project has ever had.
