# Analysis A — Direct Deep Analysis

## 1. Architecture & System Design

FinWise is a **multi-agent financial advisor** exposing its workflow through the **Model Context Protocol (MCP)**. The system is split into two .NET 10 projects with clear responsibilities:

**FinWise.McpServer** — A thin ASP.NET Core host that acts as the MCP transport layer. Program.cs serves as the **composition root** (manual DI, no container), wiring up Azure OpenAI clients, CosmosDB/in-memory stores, and the stock agent from Azure AI Foundry. It exposes exactly 2 MCP tools via FinWiseTools.cs: `run_finwise_workflow` and `reset_conversation`. McpSessionMapping.cs bridges MCP session IDs (HTTP header) to internal agent session IDs (GUIDs).

**FinWise.MultiAgentWorkflow** — A transport-agnostic class library containing all business logic. This is where the real architecture lives:

- **Hub-and-spoke orchestration**: 4 agents (Orchestrator, Profile, Advisor, Stock) connected via `AgentWorkflowBuilder.CreateHandoffBuilderWith()`. The Orchestrator is a **silent router** — it never emits user-facing text, only executes handoff tool calls. All agents route exclusively through it; no direct agent-to-agent communication.
- **Stateless agents, stateful sessions**: `AIAgent` instances are recreated per request. All state lives in `AgentSession`, persisted via the SDK's `AgentSessionStore` (from `Microsoft.Agents.AI.Hosting`). Messages are accessed via `TryGetInMemoryChatHistory()` / `SetInMemoryChatHistory()` extension methods that use the session's `StateBag`. Agent factories use `ChatClientAgentOptions` with stable `Id` properties for the SDK's composite key `{agentId}:{conversationId}`.
- **Progressive gating via `PROFILE_READY:` marker**: Until the Profile agent emits this marker into conversation history, the Orchestrator can only route to Profile. Once detected, the workflow is rebuilt with Advisor and Stock agents available. This prevents handoff loops elegantly.
- **Session reset detection**: `AgentSessionResetEvaluator.cs` uses string-matching against ~20 trigger phrases. Resets create a new agent session; user profiles (keyed by email) survive.

## 6. Improvement Recommendations

### High Priority

1. **Break up `ProcessMessageAsync`** — This 150+ line method in `FinWiseWorkflowService.cs` handles reset detection, profile gating, email augmentation, workflow execution, response validation, and persistence. Extract into a pipeline of focused steps (e.g., `DetectReset()`, `ResolveProfileAccess()`, `AugmentQuery()`, `Execute()`, `ValidateResponse()`, `Persist()`). This would improve testability dramatically.

2. **Replace string-based `PROFILE_READY:` marker with typed state** — Scanning full message history for a string marker on every request is brittle and O(n). Consider a session-level boolean flag persisted alongside the session data, updated when the Profile agent completes.

3. **~~Add a persistent `IAgentSessionStore` implementation~~** — **RESOLVED (spec 006 Phase 1 + spec 007).** The custom `IAgentSessionStore` interface was replaced with the SDK's `AgentSessionStore` abstract class. `InMemoryAgentSessionStore` (SDK built-in) is used for dev. Redis implementation planned in [spec 007](../007-redis-agent-session-store-plan.md).

### Medium Priority

4. **Reduce agent recreation** — `CreateAgentsAndWorkflow` is called up to 3 times per request (initial, post-reset, post-profile-check). Cache the workflow configurations or use a builder that doesn't require full reconstruction.

5. **Add health checks and metrics** — The `OpenTelemetry.Api` dependency exists in `Directory.Packages.props` but isn't used. Wire up traces/metrics for agent invocations, handoff counts, and session lifecycle events.

6. **Harden reset detection** — The current phrase-matching is case-insensitive but literal. "My email is" triggers a reset even in legitimate profile updates. Consider using the LLM itself to classify intent, or require a more explicit command syntax.

### Low Priority

7. **Extract MCP tool descriptions to configuration** — The 100+ word description on `run_finwise_workflow` in `FinWiseTools.cs` is instruction-injection into the MCP host's LLM. Move to a resource file for easier iteration.

8. **Add retry/circuit-breaker for Azure AI Foundry** — The stock agent resolution in `Program.cs` fails hard on startup if Foundry is unreachable. Consider lazy resolution or graceful degradation.

9. **Consider switching from `Newtonsoft.Json` to `System.Text.Json` uniformly** — The project uses both (Newtonsoft for CosmosDB legacy, STJ elsewhere). The CosmosDB SDK now supports STJ natively via `UseSystemTextJsonSerializerWithOptions`, which is already configured — the `Newtonsoft.Json` dependency may be removable.

## How It Works — Request Flow

A user message enters via `FinWiseTools.RunFinWiseWorkflow()` → extracts `MCP-Session-Id` from the HTTP header → maps it to an internal `agentSessionId` (GUID) via `McpSessionMapping` → calls `FinWiseWorkflowService.ProcessMessageAsync()`. That method: (1) creates agents with profile-only access, (2) restores the `AgentSession` from the store, (3) checks if the query matches a reset trigger, (4) scans message history for the `PROFILE_READY:` marker — if found, rebuilds the workflow with advisor+stock agents added, (5) augments standalone emails to natural language, (6) executes the handoff workflow via `InProcessExecution.RunStreamingAsync()` streaming events with a 25-invocation circuit breaker and 60s timeout, (7) validates the orchestrator didn't leak text, (8) persists the session with messages stored independently from the SDK's internal provider.

The **gating mechanism** is the key architectural insight: until `UserProfileAgent` emits `PROFILE_READY: email=... risk=... goals=... timeframe=...` into the conversation history, the `Orchestrator` can only hand off to `ProfileAgent`. This prevents the advisor from being invoked before the user is identified — and eliminates infinite handoff loops between orchestrator and advisor.

## Improvement Recommendations

### High Priority

1. **Decompose `FinWiseWorkflowService.ProcessMessageAsync()`** — This 150+ line method in `FinWiseWorkflowService.cs` does reset detection, profile gating, email augmentation, workflow execution, response validation, and persistence in one procedural flow. Extract a pipeline of focused, testable steps. This is the single biggest code quality gap.

2. **Replace `PROFILE_READY:` string scanning with persisted state** — Every request does `O(n)` scan of full message history for a string marker. Store a `bool IsProfileReady` on `AgentSessionData` instead. The marker is also fragile — any prompt change to the profile agent could break it.

3. **~~Implement persistent `IAgentSessionStore`~~** — **RESOLVED.** The custom `IAgentSessionStore` was replaced with the SDK's `AgentSessionStore` abstract class (spec 006 Phase 1). `InMemoryAgentSessionStore` is used for dev. Redis planned in [spec 007](../007-redis-agent-session-store-plan.md).

### Medium Priority

4. **Eliminate redundant agent/workflow recreation** — `CreateAgentsAndWorkflow()` is called up to 3× per request (initial build, post-reset, post-profile-gating). The gating check could determine the mode upfront rather than speculatively building and then rebuilding.

5. **Wire OpenTelemetry** — `OpenTelemetry.Api 1.12.0` is declared in `Directory.Packages.props` but unused in any `.csproj`. Add traces for agent invocations, handoff events, and CosmosDB latency.

6. **Harden `AgentSessionResetEvaluator`** — The phrase "my email is" in `AgentSessionResetEvaluator.cs` triggers a reset. But a returning user saying "my email is john@example.com" to identify themselves — not switch users — would incorrectly reset. The trigger is too broad for legitimate re-identification.

### Low Priority

7. **Remove `Newtonsoft.Json` dependency** — `FinWise.MultiAgentWorkflow.csproj` references it, but CosmosDB is already configured with `UseSystemTextJsonSerializerWithOptions`. Verify it's actually needed; if not, remove to eliminate the dual-serializer ambiguity.

8. **Add graceful degradation for stock agent** — `StockSpecializedAgentFactory.CreateAgentAsync()` is called at startup in `Program.cs`. If Azure AI Foundry is unreachable, the entire server fails to start. Consider lazy resolution or a fallback.

9. **Make tool descriptions configurable** — The `run_finwise_workflow` tool in `FinWiseTools.cs` has a 100+ word description inlined as an attribute. Move to a resource file for iteration without recompilation.
