# 008.A - Redis MCP Session Store Plan for direct MCP Session ID for Agent Session ID (No Mapping)

**Status**: Ready for Review  
**Date**: 2026-03-22  
**Depends on**: [007 — Redis AgentSessionStore](../007-redis-agent-session-store-plan/007-redis-agent-session-store-plan.md) (completed)  
**Version target**: v0.4.0  
**Variant**: A — Eliminates `McpSessionMapping` entirely; uses MCP Session ID as the agent session ID  
**Companion**: [008.B](008.B-redis-mcp-session-mapping-plan.md) — Keeps the mapping layer in Redis  

---

## 1. Summary

Enable the FinWise MCP Server to run as **multiple stateless instances** behind a load balancer — no session affinity required. This variant takes the **simplest possible approach**: eliminate the `McpSessionMapping` layer entirely and use the MCP Session ID directly as the agent session key.

**Two changes solve the problem**:
1. **`ISessionMigrationHandler` + Redis** — Upgrade the MCP SDK to 1.1.0 GA and implement session migration so the SDK's internal transport state can move between instances.
2. **Use MCP Session ID as agent session ID** — Remove the mapping layer. The `MCP-Session-Id` header value becomes the `conversationId` passed to `RedisAgentSessionStore`. Reset = clear data under the same ID instead of generating a new one.

**Result**: One fewer Redis key family, no `McpSessionMapping` class at all, simpler tool code, fewer race conditions.

## 2. Problem Analysis

_(Shared context with [008.B](008.B-redis-mcp-session-mapping-plan.md) — identical problem, different solution.)_

### 2.1 The Scale-Out Scenario

```
                          ┌─────────────────┐
MCP Client ──HTTP POST──▶ │  Load Balancer   │
(VS Code,                 │  (round-robin,   │
 Claude Desktop)          │   no affinity)   │
                          └────────┬────────┘
                        ┌──────────┼──────────┐
                        ▼          ▼          ▼
                   Instance A  Instance B  Instance C
                   (MCP Server) (MCP Server) (MCP Server)
                        │          │          │
                        └──────────┼──────────┘
                              ▼         ▼
                           Redis    CosmosDB
                        (sessions)  (profiles)
```

**What works today**: Agent sessions (Redis) and user profiles (CosmosDB) are already externalized.

**What breaks today**:

| Request # | Instance | What Happens | Result |
|-----------|----------|-------------|--------|
| 1 (initialize) | A | MCP SDK creates session `mcp-abc` in memory. `McpSessionMapping` maps `mcp-abc → agent-xyz` in memory. | ✅ Works |
| 2 (tool call) | A | SDK finds `mcp-abc` locally, mapping resolves `agent-xyz`. | ✅ Works |
| 3 (tool call) | **B** | SDK doesn't recognize `mcp-abc` → **404**. Mapping has no entry → new agent session → conversation lost. | ❌ **Broken** |

### 2.2 Two Layers of In-Memory State

| Layer | Component | Current Location | Scale-Out Impact |
|-------|-----------|-----------------|-----------------|
| **Transport** | MCP SDK session (init handshake, client capabilities) | In-memory (SDK internal) | SDK returns 404 for unknown session IDs |
| **Application** | `McpSessionMapping` (MCP session → agent session routing) | In-memory `ConcurrentDictionary` | Conversation routing lost |

### 2.3 Why the Mapping Exists Today

Looking at the current flow in `FinWiseTools.cs`:

```csharp
var agentSessionId = sessionMapping.GetOrCreateAgentSessionId(sessionId);  // mapping!
var result = await workflowService.ProcessMessageAsync(agentSessionId, query);

if (result.WasReset)
    sessionMapping.UpdateAgentSessionId(sessionId, result.AgentSessionId);  // re-mapping!
```

The mapping exists for **one reason**: session resets generate a new agent session ID (`Guid.NewGuid()`), but the MCP session stays the same (same client connection). The mapping tracks which agent session the MCP session currently points to.

**Key insight**: If resets clear the session data under the same ID instead of generating a new one, the mapping is unnecessary. The MCP Session ID itself becomes the stable agent session identifier.

**Retrospective note**: The mapping was never technically necessary — even in the original single-instance, in-memory design. It exists only because the reset logic was written as "generate new GUID" rather than "clear data, keep same key." That early design choice created a problem (MCP session outlives agent session ID), then introduced `McpSessionMapping` to solve it. If `ProcessMessageAsync` had been written from the start as "clear data, keep same ID," the `McpSessionMapping` class would never have existed. 008.A corrects this unnecessary abstraction while also solving the scale-out problem.

### 3.2 `ISessionMigrationHandler` — How It Works

```csharp
public interface ISessionMigrationHandler
{
    ValueTask OnSessionInitializedAsync(
        HttpContext context, string sessionId,
        InitializeRequestParams initializeParams,
        CancellationToken cancellationToken);

    ValueTask<InitializeRequestParams?> AllowSessionMigrationAsync(
        HttpContext context, string sessionId,
        CancellationToken cancellationToken);
}
```

**Flow across instances**:

```
1. Client → Instance A (POST initialize)
   └─ SDK creates session "mcp-abc"
   └─ OnSessionInitializedAsync("mcp-abc", initParams) → store in Redis
   └─ Tool uses "mcp-abc" directly as agent session ID → Redis

2. Client → Instance B (POST tool call with MCP-Session-Id: mcp-abc)
   └─ SDK doesn't find "mcp-abc" locally
   └─ AllowSessionMigrationAsync("mcp-abc") → reads initParams from Redis + refreshes TTL
   └─ SDK re-creates session locally
   └─ Tool uses "mcp-abc" directly as agent session ID → reads from Redis → works!
```

**Note on `DistributedCacheEventStreamStore`**: Not needed for v0.4.0. FinWise tools are request-response only (no server-initiated notifications or SSE streaming). If FinWise later uses MCP notifications/sampling, add this store for cross-instance event stream continuity.

### 3.4 Discarded Alternative: Pure Stateless MCP (`Stateless = true`)

The MCP SDK supports `HttpServerTransportOptions.Stateless = true`, which eliminates **all** server-side session state — no `MCP-Session-Id` header, no initialize handshake, no SSE channels. Each HTTP request is fully independent.

**This was evaluated and discarded for FinWise.** The fundamental problem: without `MCP-Session-Id`, the server has no way to correlate consecutive requests to the same conversation. FinWise's multi-agent workflow is inherently **multi-turn** — the orchestrator agent hands off to the profile agent to collect user information across multiple exchanges, then to the advisor agent for financial guidance. All of this state lives in an `AgentSession` keyed by a session identifier. With pure stateless MCP, every request looks identical — there is no mechanism to route a user's second message to the same conversation as their first.

Possible workarounds and why they fail:

| Workaround | Problem |
|------------|--------|
| Client passes `session_id` as a tool parameter | **Breaking change** for all MCP clients. Non-standard — violates MCP spec expectations. Clients (VS Code, Claude Desktop) would need custom logic to generate and track IDs. |
| Derive identity from authentication tokens | Requires adding an auth layer (not in scope). Would conflate _user identity_ with _session identity_ — a user can have multiple concurrent sessions. |
| Use client IP address | Unreliable behind proxies/NAT. Multiple users behind one IP would collide. |
| Single-turn only (no conversation state) | Fundamentally incompatible with FinWise's multi-agent workflow, which requires profile collection across turns before enabling advisor access. |

Additionally, `Stateless = true` disables SSE streaming, unsolicited server notifications, client sampling, elicitation, and roots capabilities. While FinWise doesn't use these today, adopting stateless mode would permanently close the door on these MCP features.

**Conclusion**: Pure stateless MCP is a non-option for any MCP server that maintains multi-turn conversations via `AgentSession`. The `ISessionMigrationHandler` approach (this plan) solves the scale-out problem while preserving the session correlation that the conversation workflow depends on.

### 3.5 Agent Session Reset — Deep Analysis

**008.A still fully supports agent session resets.** The mechanism changes from "new ID + mapping update" to "clear data + same ID" — but the user-facing behavior is identical.

**Current behavior** (in `FinWiseWorkflowService.ProcessMessageAsync`):

```csharp
// On reset:
await _sessionManager.ClearSessionAsync(agentSessionId);  // delete old session from Redis
agentSessionId = Guid.NewGuid().ToString();                // generate NEW ID
// Returns WorkflowResponse with new agentSessionId + WasReset = true
// Caller (FinWiseTools) updates McpSessionMapping to point to new ID
```

**Proposed behavior** (this variant):

```csharp
// On reset:
await _sessionManager.ClearSessionAsync(agentSessionId);  // delete session data from Redis
// KEEP the same agentSessionId — it IS the MCP Session ID
// Returns WorkflowResponse with SAME agentSessionId + WasReset = true
// Caller does NOTHING — no mapping to update
```

**Why this works**: What makes a session "fresh" is not having a new ID — it's having **empty data**. The `RedisAgentSessionStore.GetSessionAsync` already handles this:

```csharp
var json = await _redis.GetDatabase().StringGetAsync(key);
if (json.IsNullOrEmpty)
    return await agent.CreateSessionAsync(cancellationToken); // fresh session!
```

After `ClearSessionAsync` deletes the Redis key, the next request finds nothing and creates a fresh session. The ID is just a storage key — an empty key slot = fresh start.

**Full reset lifecycle under 008.A**:

```
1. User says "hello" → fresh AgentSession under mcp-abc → asks for email
2. User provides "adran@example.com" → profile loaded → PROFILE_READY
3. User asks financial questions → advisor responds
4. User says "reset" → ClearSessionAsync("mcp-abc") → Redis key deleted
5. User says "hello" → GetSessionAsync("mcp-abc") → not found → fresh session
6. No PROFILE_READY → orchestrator routes to profile agent → asks for email
7. User provides different email → different profile from CosmosDB
```

**What's preserved**:
- `reset_conversation` tool: still works, simpler (no mapping update, no new ID)
- `WasReset` flag: still fires, still triggers the "session reset" response message
- Orchestrator's `request_session_reset` tool: still works mid-conversation
- User profiles in CosmosDB: untouched by reset (keyed by email, not by session ID)
- `mcpinit:*` Redis key: untouched — the MCP transport session stays alive

**What's removed**: The `Guid.NewGuid()` call and the mapping update. Zero user-visible change.

### 3.6 Multi-Client Parallel Sessions

A single user (e.g., `adrian@outlook.com`) can use **multiple MCP clients simultaneously**, each with a completely independent conversation:

```
VS Code (Copilot)  → MCP Session: mcp-abc  → Agent Session: mcp-abc
ChatGPT            → MCP Session: mcp-xyz  → Agent Session: mcp-xyz
Claude Desktop     → MCP Session: mcp-def  → Agent Session: mcp-def
```

Each MCP client creates its own MCP session during `initialize`. Different sessions = different Redis keys = **fully isolated conversations**. Each client has its own chat history, its own `PROFILE_READY` progression, and its own position in the multi-agent workflow.

**What IS shared**: The CosmosDB user profile. All three clients ask for email, the user types `adrian@outlook.com` in each, and all three load the same profile. That's correct — the profile represents the user's identity and preferences, not the conversation.

```
Redis:
  agentsession:orchestrator_agent:mcp-abc  →  [VS Code conversation]
  agentsession:orchestrator_agent:mcp-xyz  →  [ChatGPT conversation]
  agentsession:orchestrator_agent:mcp-def  →  [Claude conversation]

CosmosDB:
  adrian@outlook.com  →  [single shared profile]
```

**Reset is per-client**: Resetting in VS Code clears `mcp-abc`'s data only. The ChatGPT and Claude conversations continue independently.

This is a natural strength of 008.A — the 1:1 mapping between MCP session and conversation means parallel clients are automatically isolated without any extra logic.

### 3.7 In-Memory Store Compatibility — Reset Caveat

Under the current design, `AgentSessionManager.ClearSessionAsync` delegates to `IClearableSessionStore` — but the SDK's `InMemoryAgentSessionStore` does **not** implement that interface. Clearing is a no-op:

```csharp
if (_sessionStore is IClearableSessionStore clearable)
    await clearable.ClearSessionAsync(agentSessionId);  // Redis: deletes the key
else
    Log.Debug("no-op for in-memory store");              // InMemory: does nothing!
```

This creates an asymmetry between 008.A and the current approach:

| | Current (new GUID on reset) | 008.A (clear data, keep same ID) |
|---|---|---|
| **Redis store** | Delete key + new GUID → fresh ✅ | Delete key + same ID → `GetSessionAsync` finds nothing → fresh ✅ |
| **In-memory store** | No-op clear + new GUID → `GetSessionAsync` finds nothing for new GUID → fresh ✅ (by accident) | No-op clear + same ID → `GetSessionAsync` **finds old data** → **reset broken** ❌ |

The current approach works with in-memory *by accident* — the orphaned session stays in memory forever (a leak), but the new GUID has no data so it looks fresh. Under 008.A, keeping the same ID means the stale data persists.

**Potential future fix** (NOT in scope for v0.4.0): A custom `ClearableInMemoryAgentSessionStore` that implements both `AgentSessionStore` and `IClearableSessionStore` would solve this. It would be a small `ConcurrentDictionary` wrapper with a `Remove` method — the same pattern as `RedisAgentSessionStore` but in-memory. This would also fix the existing memory leak where orphaned sessions accumulate on every reset.

**For v0.4.0, we continue using the SDK's `InMemoryAgentSessionStore`** from the Microsoft Agent Framework. The practical impact is limited:
- In-memory mode is used only for local development (single instance)
- Session reset in in-memory mode will **not fully clear the session** — the stale data remains in memory under the same MCP Session ID
- **Workaround**: Restart the MCP server connection (Stop/Start in VS Code, or restart the server process) to get a truly fresh session
- Once Redis is enabled (the production path), reset works correctly via `RedisAgentSessionStore.ClearSessionAsync`

This is an acceptable trade-off: the in-memory fallback is a convenience for development, not a production path. The `ClearableInMemoryAgentSessionStore` can be added in a future version if clean resets in no-Redis mode become important.

## 4. Detailed Design

### 4.1 Component Overview

```
  MCP Client
      │ HTTP + MCP-Session-Id header
      ▼
┌─────────────────────────────────────────────────────┐
│ Any FinWise.McpServer Instance                      │
│                                                     │
│  MCP SDK (Stateless=false, with ISessionMigration)  │
│    ├─ OnSessionInitializedAsync → Redis             │
│    └─ AllowSessionMigrationAsync ← Redis            │
│                                                     │
│  FinWiseTools                                       │
│    └─ Uses MCP-Session-Id directly as agentSessionId│
│       (NO mapping layer)                            │
│                                                     │
│  FinWiseWorkflowService (stateless)                 │
│    └─ AgentSessionManager → RedisAgentSessionStore  │
└─────────────────────────────────────────────────────┘
      │              │
      ▼              ▼
   Redis          CosmosDB
  (init params)   (profiles)
  (sessions)
```

**Key difference from 008.B**: No `mcpsession:*` Redis keys. No `McpSessionMapping` class. No `IMcpSessionMapping` interface.

### 4.2 Redis Key Design

| Key Pattern | Value | TTL | Purpose |
|-------------|-------|-----|---------|
| `mcpinit:{sessionId}` | JSON-serialized `InitializeRequestParams` | 24h sliding | MCP SDK session migration |
| `agentsession:{agentId}:{conversationId}` | Serialized `AgentSession` JSON | 24h sliding (existing) | Agent conversation state |

**Compared to 008.B**: One fewer key family (`mcpsession:*` eliminated). The `conversationId` in `agentsession:*` keys is now the MCP Session ID string instead of a GUID.

### 4.3 File-by-File Changes

#### 4.3.1 `RedisSessionMigrationHandler.cs` (New)

**Location**: `src/FinWise.McpServer/Infrastructure/McpSession/Redis/RedisSessionMigrationHandler.cs`  
**Responsibility**: Implements `ISessionMigrationHandler` to persist/restore MCP init handshake data in Redis.

```csharp
public sealed class RedisSessionMigrationHandler : ISessionMigrationHandler
{
    private readonly IConnectionMultiplexer _redis;
    private readonly TimeSpan _ttl;

    public RedisSessionMigrationHandler(IConnectionMultiplexer redis, TimeSpan ttl)
    {
        _redis = redis;
        _ttl = ttl;
    }

    public async ValueTask OnSessionInitializedAsync(
        HttpContext context, string sessionId,
        InitializeRequestParams initializeParams,
        CancellationToken cancellationToken)
    {
        var json = JsonSerializer.Serialize(initializeParams);
        await _redis.GetDatabase().StringSetAsync(
            GetKey(sessionId), json, _ttl);
    }

    public async ValueTask<InitializeRequestParams?> AllowSessionMigrationAsync(
        HttpContext context, string sessionId,
        CancellationToken cancellationToken)
    {
        var key = GetKey(sessionId);
        var json = await _redis.GetDatabase().StringGetAsync(key);
        if (json.IsNullOrEmpty) return null;

        // Refresh TTL on successful migration — makes this a true sliding window.
        // Without this, a session initialized at hour 0 could expire at hour 24
        // even if actively migrating between instances.
        await _redis.GetDatabase().KeyExpireAsync(key, _ttl);

        return JsonSerializer.Deserialize<InitializeRequestParams>(json!);
    }

    internal static string GetKey(string sessionId) => $"mcpinit:{sessionId}";
}
```

#### 4.3.3 `McpSessionAccessor.cs` (New — extracted static method)

The static `GetSessionId(HttpContext)` method moves from `McpSessionMapping` to a standalone static class at `Infrastructure/McpSession/McpSessionAccessor.cs`, since `McpSessionMapping` is being deleted:

```csharp
public static class McpSessionAccessor
{
    public static string GetSessionId(HttpContext httpContext)
    {
        if (httpContext.Request.Headers.TryGetValue("MCP-Session-Id", out var sessionHeader)
            && !StringValues.IsNullOrEmpty(sessionHeader))
        {
            return sessionHeader.ToString();
        }
        throw new InvalidOperationException("Missing MCP-Session-Id header.");
    }
}
```

#### 4.3.4 `McpSessionMapping.cs` (Deleted)

The entire class is removed. No interface, no in-memory implementation, no Redis implementation.

#### 4.3.5 `FinWiseWorkflowService.cs` (Modified — reset behavior)

**Current reset logic** (inside `ProcessMessageAsync`):
```csharp
if (wasReset)
{
    await _sessionManager.ClearSessionAsync(agentSessionId);

```

**Proposed reset logic**:
```csharp
if (wasReset)
{

```

**Current**:
```csharp
public async Task<string> ResetSessionAsync(string agentSessionId)
{

    var newAgentSessionId = Guid.NewGuid().ToString();
    return newAgentSessionId;
}
```

**Proposed**:
```csharp
public async Task ResetSessionAsync(string agentSessionId)
{
    await _sessionManager.ClearSessionAsync(agentSessionId);
    // No new ID — caller continues using the same ID
}
```

Return type changes from `Task<string>` to `Task`. The caller doesn't need a new ID.

### 4.4 Agent Session ID Format — Before and After

| Scenario | Before (GUID) | After (MCP Session ID) |
|----------|--------------|----------------------|
| Redis key example | `agentsession:orchestrator_agent:a1b2c3d4-...` | `agentsession:orchestrator_agent:mcp-session-abc123...` |
| Key length | ~70 chars | ~80–90 chars (MCP session IDs are longer) |
| ID stability across reset | New GUID on every reset | Same ID, session data cleared |
| Format control | We generate it | MCP SDK generates it |

**Risk assessment**: The `RedisAgentSessionStore.GetKey` method uses the ID as a plain string component (`agentsession:{agentId}:{conversationId}`). It doesn't validate format. MCP Session IDs are opaque strings — safe for Redis keys.

## 8. Future Authentication Analysis

### 8.1 MCP OAuth and Session Independence

The MCP specification (2025-03-26) defines OAuth 2.1 authorization as an **optional, per-request concern** that is independent of MCP sessions. From the spec:

> *"authorization MUST be included in every HTTP request from client to server, even if they are part of the same logical session."*

This means the `Authorization: Bearer <token>` header and the `MCP-Session-Id` header are **orthogonal** — changing the authenticated user (different Bearer token) does NOT change the MCP session. A single MCP session can carry requests from different users if the client swaps tokens.

### 8.2 Does This Break 008.A?

**No — it's handled by composite keys.** If FinWise adds authentication in the future, the tool code can derive the agent session ID from both the MCP session and the authenticated user:

```csharp
var mcpSessionId = McpSessionAccessor.GetSessionId(httpContext);
var userId = httpContext.User.FindFirst("sub")?.Value; // from JWT
var agentSessionId = userId != null
    ? $"{mcpSessionId}:{userId}"   // auth enabled: user-scoped
    : mcpSessionId;                // no auth: MCP session-scoped (today)
```

This produces keys like `agentsession:orchestrator_agent:mcp-abc:user123` — naturally separated per user, per MCP session. No mapping layer needed. No Redis lookup for routing. The composite key IS the routing.

### 8.3 When Would You Actually Need a Mapping Layer?

A mapping (as in [008.B](008.B-redis-mcp-session-mapping-plan.md)) adds value only when the relationship between (MCP session + user identity) and agent session is **not a simple derivable function** — i.e., the agent session ID needs to be independently mutable:

| Scenario | Composite Key (008.A) | Mapping (008.B) |
|----------|:---:|:---:|
| 1 user, 1 conversation per MCP session | ✅ | ✅ |
| Auth: different users on same MCP session | ✅ (composite key) | ✅ |
| "Fork" a conversation (clone session, continue both) | ❌ | ✅ |
| Multiple concurrent conversations per user per session | ❌ | ✅ |
| Admin "takes over" another user's session | ❌ | ✅ |

FinWise's model is strictly 1:1 — one conversation per user per MCP session. None of the fork/multi-conversation scenarios are in scope or planned. The composite key approach handles all realistic auth scenarios without a mapping layer.

### 8.4 VS Code Session Lifecycle

Actions that create a **new MCP session** (new `MCP-Session-Id`):
- Closing and reopening VS Code
- Stopping and starting the MCP server (right-click → Stop/Start, or code lenses in `mcp.json`)
- Editing `mcp.json` (VS Code auto-restarts the server)
- `MCP: Restart Server` command
- Server idle timeout (30 min in FinWise) → client reconnects
- `Developer: Reload Window`

Actions that keep the **same MCP session**:
- Opening new chat panels in the same VS Code window
- Multiple Copilot Chat conversations in the same window
- Changing the `Authorization` header (different user token) — same session, different auth

For single-user VS Code, 008.A works naturally: one VS Code window = one MCP session = one conversation. For future multi-user scenarios with auth, the composite key approach handles it cleanly.
