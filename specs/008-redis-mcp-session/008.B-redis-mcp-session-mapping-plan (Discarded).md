# 008.B — Redis MCP Session Store Plan with Mapping between MCP Session ID and Agent Session ID

**Status**: Ready for Review  
**Date**: 2026-03-22  
**Depends on**: [007 — Redis AgentSessionStore](../007-redis-agent-session-store-plan/007-redis-agent-session-store-plan.md) (completed)  
**Version target**: v0.4.0  
**Variant**: B — Keeps the `McpSessionMapping` layer, moves it to Redis  
**Companion**: [008.A](008.A-redis-mcp-session-store-plan.md) — Eliminates the mapping entirely  

---

## 1. Summary

Enable the FinWise MCP Server to run as **multiple stateless instances** behind a load balancer — no session affinity (sticky sessions) required. This variant preserves the existing **mapping layer** between MCP session IDs and agent session IDs, moving it from an in-memory `ConcurrentDictionary` to Redis.

**Two changes solve the problem**:
1. **`ISessionMigrationHandler` + Redis** — Upgrade the MCP SDK to 1.1.0 GA and implement session migration so the SDK's internal transport state can move between instances.
2. **`McpSessionMapping` → Redis** — Extract an `IMcpSessionMapping` interface with in-memory and Redis implementations. The mapping between MCP session IDs and agent session IDs moves to Redis, accessible from any instance.

**Result**: MCP sessions and agent sessions keep separate identities (as today). The current reset behavior (generate new agent session ID) is preserved. More moving parts than 008.A, but no changes to the workflow layer.

## 2. Problem Analysis

_(Shared context with [008.A](008.A-redis-mcp-session-store-plan.md) — identical problem, different solution.)_

### 2.1 The Scale-Out Scenario

```
                          ┌─────────────────┐
MCP Client ──HTTP POST──▶ │  Load Balancer   │
(VS Code,                 │  (round-robin,   │
 Claude Desktop)          │   no affinity)   │
                          └────────┬────────┘
                        ┌──────────┼──────────┐
                        ▼          ▼          ▼
                   Instance A  Instance B  Instance C
                   (MCP Server) (MCP Server) (MCP Server)
                        │          │          │
                        └──────────┼──────────┘
                              ▼         ▼
                           Redis    CosmosDB
                        (sessions)  (profiles)
```

**What works today**: Agent sessions (Redis) and user profiles (CosmosDB) are already externalized — any instance can read/write them.

**What breaks today**:

| Request # | Instance | What Happens | Result |
|-----------|----------|-------------|--------|
| 1 (initialize) | A | MCP SDK creates session `mcp-abc`, stores in-memory. `McpSessionMapping` maps `mcp-abc → agent-xyz`. | ✅ Works |
| 2 (tool call) | A | MCP SDK finds `mcp-abc` locally, `McpSessionMapping` resolves `agent-xyz`. | ✅ Works |
| 3 (tool call) | **B** | MCP SDK doesn't recognize `mcp-abc` → **404**. Even if it did, `McpSessionMapping` has no entry → **new agent session** → conversation lost. | ❌ **Broken** |

### 2.2 Two Distinct Layers of State

The problem exists at **two independent layers**, each requiring its own solution:

**Layer 1 — Application State: `McpSessionMapping`**

| Aspect | Current | Target |
|--------|---------|--------|
| What | Maps MCP session IDs → agent session IDs | Same mapping |
| Where | In-memory `ConcurrentDictionary` | Redis |
| Impact | Conversation routing lost on instance switch | Any instance can resolve the mapping |
| Pattern | Follow `RedisAgentSessionStore` | `mcpsession:{mcpSessionId}` key → agent session ID value |

**Layer 2 — Transport State: MCP SDK Internal Session**

| Aspect | Current | Target |
|--------|---------|--------|
| What | MCP protocol handshake, client capabilities, protocol version | Same data, externalized |
| Where | In-memory (SDK manages internally) | Redis (via `ISessionMigrationHandler`) |
| Impact | SDK returns 404 for unrecognized session IDs | SDK re-creates session from persisted init params |
| Solution | Not available in SDK 0.5.0-preview.1 | `ISessionMigrationHandler` (added in SDK 0.9.0-preview.1) |

### 3.2 `ISessionMigrationHandler` — How It Works

```csharp
public interface ISessionMigrationHandler
{
    // Called after successful initialize handshake — persist to external store
    ValueTask OnSessionInitializedAsync(
        HttpContext context,
        string sessionId,
        InitializeRequestParams initializeParams,
        CancellationToken cancellationToken);

    // Called when request has unrecognized session ID — return init params to migrate,
    // or null to reject (404)
    ValueTask<InitializeRequestParams?> AllowSessionMigrationAsync(
        HttpContext context,
        string sessionId,
        CancellationToken cancellationToken);
}
```

**Flow when deployed across multiple instances**:

```
1. Client → Instance A (POST initialize)
   └─ SDK creates session "mcp-abc"
   └─ OnSessionInitializedAsync("mcp-abc", initParams) → store in Redis
   └─ McpSessionMapping.GetOrCreateAgentSessionId("mcp-abc") → store in Redis

2. Client → Instance B (POST tool call with MCP-Session-Id: mcp-abc)
   └─ SDK doesn't find "mcp-abc" locally
   └─ AllowSessionMigrationAsync("mcp-abc") → reads initParams from Redis → returns them
   └─ SDK re-creates session "mcp-abc" locally with client state pre-populated
   └─ McpSessionMapping.GetAgentSessionId("mcp-abc") → reads from Redis → "agent-xyz"
   └─ FinWiseWorkflowService.ProcessMessageAsync("agent-xyz", query) → works!
```

**Important limitation from SDK docs**: `ISessionMigrationHandler` does NOT solve in-flight server-to-client requests (sampling, elicitation). Responses to those must still route to the originating process. For FinWise, this is a non-issue — we don't use sampling, elicitation, or roots capabilities.

### 3.3 `Stateless = true` Mode — Why It Doesn't Work for FinWise

When `Stateless = true`:
- `McpSession.SessionId` is `null` — no `MCP-Session-Id` header
- `RunSessionHandler` called once per request (session per request)
- `/sse` GET endpoint disabled
- Unsolicited server-to-client messages unsupported
- Client sampling, elicitation, and roots capabilities disabled
- **Zero server-side MCP session state** — perfect for load balancing

**The fatal flaw**: Without `MCP-Session-Id`, the server has no way to correlate requests to an ongoing conversation. FinWise's multi-agent workflow is inherently multi-turn — profile collection alone requires several back-and-forth exchanges before the advisor can be unlocked. Every request would look identical, with no mechanism to route a user's second message to the same `AgentSession` as their first. This is analyzed in detail in Section 4 ("Discarded: `Stateless = true`").

### 3.4 StackExchange.Redis — Already in Use

**Version in repo**: `2.12.4` (GA, mature)  
Already used for `RedisAgentSessionStore`. The same `IConnectionMultiplexer` singleton can be shared — no new packages needed.

### 3.5 What Capabilities Does FinWise Actually Use?

Auditing `FinWiseTools.cs` against MCP capabilities:

| MCP Capability | Used by FinWise? | Impact of Losing It |
|---------------|------------------|---------------------|
| Stateful sessions (`MCP-Session-Id`) | ✅ Yes — routes to agent sessions | Must find alternative routing |
| Tool calls (POST) | ✅ Yes — core functionality | Works in both modes |
| SSE streaming (GET) | ❌ No | None |
| Unsolicited notifications | ❌ No | None |
| Client sampling | ❌ No | None |
| Elicitation | ❌ No | None |
| Roots | ❌ No | None |
| Session DELETE | ❌ No (reset via tool) | None |

**Conclusion**: FinWise only uses tool calls and session identification — but session identification is non-negotiable. The `MCP-Session-Id` header is how FinWise correlates requests to multi-turn agent conversations. Disabling it (`Stateless = true`) would break the core workflow.

### 3.6 Agent Session Reset Under Both Variants

Both 008.A and 008.B fully support agent session resets. The mechanism differs but the user-facing behavior is identical:

| Aspect | 008.A (No Mapping) | 008.B (With Mapping) |
|--------|-------------------|---------------------|
| Reset action | `ClearSessionAsync(mcpSessionId)` — deletes Redis key | `ClearSessionAsync(agentSessionId)` — deletes Redis key, generates new GUID, updates mapping |
| Agent session ID after reset | Same (MCP Session ID) | New (fresh GUID) |
| Next request | `GetSessionAsync` finds empty key → fresh session | `GetSessionAsync` finds no key for new GUID → fresh session |
| User experience | Asked for email again, can switch users | Asked for email again, can switch users |
| Profile in CosmosDB | Untouched (keyed by email) | Untouched (keyed by email) |
| `reset_conversation` tool | Simpler (no mapping update) | Current behavior preserved |
| `WasReset` flag | Still fires | Still fires |

Both approaches result in the same user experience: conversation history cleared, orchestrator asks for email, user can provide a different email to load a different profile.

### 3.7 Multi-Client Parallel Sessions

A single user (e.g., `adrian@outlook.com`) can use **multiple MCP clients simultaneously**, each with a completely independent conversation. This works identically under both 008.A and 008.B:

```
VS Code (Copilot)  → MCP Session: mcp-abc  → Agent Session (own conversation)
ChatGPT            → MCP Session: mcp-xyz  → Agent Session (own conversation)
Claude Desktop     → MCP Session: mcp-def  → Agent Session (own conversation)
```

Each MCP client creates its own MCP session during `initialize`. Different sessions = different agent sessions (whether via direct ID in 008.A or via mapping in 008.B) = **fully isolated conversations**.

**What IS shared across clients**: The CosmosDB user profile. All three clients ask for email, the user provides `adrian@outlook.com`, and all three load the same profile (investment preferences, risk tolerance, etc.). This is correct — the profile represents the user's identity, not the conversation.

**Reset is per-client**: Resetting in VS Code clears only that client's conversation. ChatGPT and Claude conversations continue independently.

## 4. Approach Options

### Option A — `ISessionMigrationHandler` + Redis (Recommended)

**Keep `Stateless = false`**, upgrade MCP SDK to 1.1.0, implement `ISessionMigrationHandler` backed by Redis.

| Aspect | Details |
|--------|---------|
| **MCP SDK upgrade** | `0.5.0-preview.1` → `1.1.0` (GA) |
| **New interface implementation** | `RedisSessionMigrationHandler : ISessionMigrationHandler` |
| **McpSessionMapping** | New `RedisMcpSessionMapping` replacing in-memory version |
| **Redis keys** | `mcpinit:{sessionId}` for init params, `mcpsession:{sessionId}` for agent mapping |
| **Existing behavior** | 100% preserved — clients see no difference |
| **MCP features** | All preserved (SSE, notifications, etc. — even if unused today) |
| **Complexity** | Moderate — SDK upgrade + 2 new Redis-backed classes |

**Pros**:
- SDK-prescribed solution for horizontal scaling (official API, merged to GA)
- No behavioral change for MCP clients — transparent migration
- Preserves all MCP capabilities for future use
- MCP SDK moves from preview to stable (production-ready)
- Pattern already proven in codebase (`RedisAgentSessionStore`)

**Cons**:
- MCP SDK major version upgrade with breaking changes (contained, but requires testing)
- Two new Redis stores to maintain (init params + session mapping)
- `ISessionMigrationHandler` only supports idle session migration — active SSE streams can't migrate mid-flight (not an issue for FinWise)

### Discarded: `Stateless = true` (Pure Stateless MCP)

The MCP SDK supports `HttpServerTransportOptions.Stateless = true`, which eliminates **all** server-side session state — no `MCP-Session-Id` header, no initialize handshake, no SSE channels. Each HTTP request is fully independent.

**This was evaluated and discarded for FinWise.** The fundamental problem: without `MCP-Session-Id`, the server has no way to correlate consecutive requests to the same conversation. FinWise's multi-agent workflow is inherently **multi-turn** — the orchestrator agent hands off to the profile agent to collect user information across multiple exchanges, then to the advisor agent for financial guidance. All of this state lives in an `AgentSession` keyed by a session identifier. With pure stateless MCP, every request looks identical — there is no mechanism to route a user's second message to the same conversation as their first.

Possible workarounds and why they fail:

| Workaround | Problem |
|------------|--------|
| Client passes `session_id` as a tool parameter | **Breaking change** for all MCP clients. Non-standard — violates MCP spec expectations. Clients (VS Code, Claude Desktop) would need custom logic to generate and track IDs. |
| Derive identity from authentication tokens | Requires adding an auth layer (not in scope). Would conflate _user identity_ with _session identity_ — a user can have multiple concurrent sessions. |
| Use client IP address | Unreliable behind proxies/NAT. Multiple users behind one IP would collide. |
| Single-turn only (no conversation state) | Fundamentally incompatible with FinWise's multi-agent workflow, which requires profile collection across turns before enabling advisor access. |

Additionally, `Stateless = true` disables SSE streaming, unsolicited server notifications, client sampling, elicitation, and roots capabilities. While FinWise doesn't use these today, adopting stateless mode would permanently close the door on these MCP features.

**Conclusion**: Pure stateless MCP is a non-option for any MCP server that maintains multi-turn conversations via `AgentSession`. It is not considered a viable approach for FinWise.

### Discarded: Sticky Sessions (ARR Affinity)

**Keep everything in-memory**, use Azure ARR affinity to pin clients to instances.

| Aspect | Details |
|--------|---------|
| **Code changes** | None |
| **Infrastructure** | Azure App Service / Container Apps with ARR affinity enabled |
| **Redis keys** | None new |
| **Complexity** | Zero code, some infrastructure config |

**Pros**:
- No code changes at all
- Works immediately
- Proven pattern for stateful web apps

**Cons**:
- **Not truly stateless** — defeats the purpose of horizontal scaling
- Uneven load distribution — popular clients pin to one instance
- Instance failure = session loss (no failover)
- ARR affinity adds latency (cookie-based routing)
- Doesn't work with all load balancer types (e.g., L4 TCP balancers)
- Blocks future move to serverless/event-driven architectures

### Comparison Matrix

| Criterion | A: SessionMigration | Stateless Mode | Sticky Sessions |
|-----------|:------------------:|:--------------:|:---------------:|
| Truly stateless instances | ✅ | ✅ | ❌ |
| No client-side changes | ✅ | ❌ | ✅ |
| MCP features preserved | ✅ | ❌ | ✅ |
| Instance failure resilience | ✅ | ✅ | ❌ |
| Supports multi-turn conversations | ✅ | ❌ | ✅ |
| Even load distribution | ✅ | ✅ | ❌ |
| Future-proof for MCP evolution | ✅ | ❌ | ❌ |

### Recommendation

**`ISessionMigrationHandler` + Redis** is the only viable approach for FinWise.

Pure stateless MCP was discarded because FinWise requires multi-turn conversation correlation — there is no standard mechanism to identify a client without `MCP-Session-Id`. Sticky sessions were discarded because they provide no instance failure resilience, create uneven load distribution, and don't represent true statelessness.

The `ISessionMigrationHandler` approach solves the scale-out problem at the infrastructure level while preserving the standard MCP protocol contract that FinWise's multi-agent conversation workflow depends on.

## 5. Recommended Approach — Detailed Design

### 5.1 Component Overview

```
  MCP Client
      │ HTTP + MCP-Session-Id header
      ▼
┌─────────────────────────────────────────────────────┐
│ Any FinWise.McpServer Instance                      │
│                                                     │
│  MCP SDK (Stateless=false, with ISessionMigration)  │
│    ├─ OnSessionInitializedAsync → Redis             │
│    └─ AllowSessionMigrationAsync ← Redis            │
│                                                     │
│  RedisMcpSessionMapping                             │
│    ├─ GetOrCreateAgentSessionId → Redis             │
│    └─ TryGetAgentSessionId ← Redis                  │
│                                                     │
│  FinWiseWorkflowService (stateless)                 │
│    └─ AgentSessionManager → RedisAgentSessionStore  │
└─────────────────────────────────────────────────────┘
      │              │              │
      ▼              ▼              ▼
   Redis          Redis          CosmosDB
  (init params)  (sessions)     (profiles)
  (mapping)
```

### 5.2 Redis Key Design

All FinWise Redis keys should be namespaced to avoid collisions:

| Key Pattern | Value | TTL | Purpose |
|-------------|-------|-----|---------|
| `mcpinit:{sessionId}` | JSON-serialized `InitializeRequestParams` | Same as session (24h sliding) | MCP SDK session migration |
| `mcpsession:{mcpSessionId}` | Agent session ID string (GUID) | Same as session (24h sliding) | MCP → agent session routing |
| `agentsession:{agentId}:{conversationId}` | Serialized `AgentSession` JSON | 24h sliding (existing) | Agent conversation state |

**TTL strategy**: All three key families use the same 24-hour sliding TTL (`SessionTtlMinutes` from `RedisOptions`). This ensures they expire together — a dangling mapping to an expired session is harmless but wasteful.

### 5.3 New/Modified Components

#### 5.3.1 `RedisMcpSessionMapping` (New — replaces `McpSessionMapping`)

**Location**: `src/FinWise.McpServer/RedisMcpSessionMapping.cs`  
**Responsibility**: Same as current `McpSessionMapping` but backed by Redis.

```csharp
// Interface extracted from McpSessionMapping for abstraction
public interface IMcpSessionMapping
{
    static abstract string GetSessionId(HttpContext httpContext);
    string GetOrCreateAgentSessionId(string sessionId);
    string? TryGetAgentSessionId(string sessionId);
    void UpdateAgentSessionId(string sessionId, string newAgentSessionId);
}
```

Wait — the current `McpSessionMapping` has a static method `GetSessionId` and instance methods for mapping. The static method extracts the MCP-Session-Id from the HTTP header and doesn't need Redis. The instance methods do the mapping.

**Design decision**: Extract an interface `IMcpSessionMapping` for the instance methods. Keep the static `GetSessionId` as a static helper (or move to a separate utility). The interface enables swapping in-memory ↔ Redis implementations.

Actually, given that `GetSessionId` is a pure static function (extracts header from `HttpContext`), it should remain static. The interface covers only the mapping operations:

```csharp
public interface IMcpSessionMapping
{
    string GetOrCreateAgentSessionId(string sessionId);
    string? TryGetAgentSessionId(string sessionId);
    void UpdateAgentSessionId(string sessionId, string newAgentSessionId);
}
```

The Redis implementation uses `IDatabase.StringGetAsync` / `StringSetAsync`:

```
Key: mcpsession:{mcpSessionId}  →  Value: {agentSessionId}
```

**Note**: `GetOrCreateAgentSessionId` must be atomic — use a Redis Lua script or `StringSetAsync` with `When.NotExists` to avoid race conditions where two instances both try to create a mapping for the same MCP session ID.

#### 5.3.2 `RedisSessionMigrationHandler` (New)

**Location**: `src/FinWise.McpServer/RedisSessionMigrationHandler.cs`  
**Responsibility**: Implements `ISessionMigrationHandler` to persist/restore MCP init handshake data in Redis.

```
Key: mcpinit:{sessionId}  →  Value: JSON-serialized InitializeRequestParams
```

- `OnSessionInitializedAsync`: Serialize `InitializeRequestParams` to JSON, store in Redis with TTL
- `AllowSessionMigrationAsync`: Look up session ID in Redis, deserialize and return params (or null if not found)

#### 5.3.3 `McpSessionMapping` refactored (Existing → interface extraction)

The existing `McpSessionMapping` becomes the in-memory implementation of `IMcpSessionMapping`. A new `RedisMcpSessionMapping` provides the Redis implementation. Both implementations are swappable via the `Redis:Enabled` toggle.

| Config | Implementation |
|--------|---------------|
| `Redis:Enabled = false` | `McpSessionMapping` (in-memory `ConcurrentDictionary`, existing behavior) |
| `Redis:Enabled = true` | `RedisMcpSessionMapping` (Redis-backed) |

#### 5.3.4 `Program.cs` Changes

```csharp
// Replace:
builder.Services.AddSingleton(new McpSessionMapping());

// With:
if (redisOptions.Enabled)
{
    builder.Services.AddSingleton<IMcpSessionMapping>(
        new RedisMcpSessionMapping(redis, TimeSpan.FromMinutes(redisOptions.SessionTtlMinutes)));
    builder.Services.AddSingleton<ISessionMigrationHandler>(
        new RedisSessionMigrationHandler(redis, TimeSpan.FromMinutes(redisOptions.SessionTtlMinutes)));
}
else
{
    builder.Services.AddSingleton<IMcpSessionMapping>(new McpSessionMapping());
}
```

`ISessionMigrationHandler` registered in DI is automatically discovered by the MCP SDK (as documented in `HttpServerTransportOptions.SessionMigrationHandler` — "If this property is not set, the server will attempt to resolve an `ISessionMigrationHandler` from DI").

#### 5.3.5 `FinWiseTools.cs` Changes

Replace `McpSessionMapping` with `IMcpSessionMapping`:

```csharp
var sessionMapping = serviceProvider.GetRequiredService<IMcpSessionMapping>();
```

The static `GetSessionId(httpContext)` call remains unchanged.

### 5.5 Async API Design

The current `McpSessionMapping` has synchronous methods (`GetOrCreateAgentSessionId`, `TryGetAgentSessionId`, `UpdateAgentSessionId`). Redis operations are inherently async.

**Design choice**: Make the `IMcpSessionMapping` interface async:

```csharp
public interface IMcpSessionMapping
{
    Task<string> GetOrCreateAgentSessionIdAsync(string sessionId);
    Task<string?> TryGetAgentSessionIdAsync(string sessionId);
    Task UpdateAgentSessionIdAsync(string sessionId, string newAgentSessionId);
}
```

The in-memory implementation can use `Task.FromResult` for the async methods. The Redis implementation uses native async Redis operations.

`FinWiseTools.cs` methods are already `async Task<string>`, so the await propagation is natural.

## 10. Future Authentication Analysis

### 10.1 MCP OAuth and Session Independence

The MCP specification (2025-03-26) defines OAuth 2.1 authorization as an **optional, per-request concern** that is independent of MCP sessions. From the spec:

> *"authorization MUST be included in every HTTP request from client to server, even if they are part of the same logical session."*

This means the `Authorization: Bearer <token>` header and the `MCP-Session-Id` header are **orthogonal** — changing the authenticated user (different Bearer token) does NOT change the MCP session. A single MCP session can carry requests from different users if the client swaps tokens.

### 10.2 How 008.B Handles Auth — And Whether It's Necessary

With the mapping layer, auth integration is straightforward — extend the mapping key to include the user:

```csharp
var agentSessionId = sessionMapping.GetOrCreateAgentSessionIdAsync(mcpSessionId, authenticatedUserId);
```

However, **the mapping layer is not required for this**. A simpler composite key approach achieves the same result without the mapping:

```csharp
var agentSessionId = $"{mcpSessionId}:{userId}";
```

This produces keys like `agentsession:orchestrator_agent:mcp-abc:user123` — naturally separated per user, per MCP session, with zero Redis lookups for routing. This is the approach [008.A](008.A-redis-mcp-session-store-plan.md) would use.

### 10.3 When Does the Mapping Layer Actually Add Value Over Composite Keys?

The mapping provides real advantages only when the relationship between (MCP session + user identity) and agent session is **not a simple derivable function** — i.e., the agent session ID needs to be independently mutable:

| Scenario | Composite Key (008.A) | Mapping (008.B) |
|----------|:---:|:---:|
| 1 user, 1 conversation per MCP session | ✅ | ✅ |
| Auth: different users on same MCP session | ✅ (composite key) | ✅ |
| "Fork" a conversation (clone session, continue both) | ❌ | ✅ |
| Multiple concurrent conversations per user per session | ❌ | ✅ |
| Admin "takes over" another user's session | ❌ | ✅ |

FinWise's model is strictly 1:1 — one conversation per user per MCP session. None of the fork/multi-conversation scenarios are in scope or planned. This means the mapping layer's auth benefit over 008.A's composite key approach is **marginal for FinWise's use case**.

### 10.4 VS Code Session Lifecycle

Actions that create a **new MCP session** (new `MCP-Session-Id`):
- Closing and reopening VS Code
- Stopping and starting the MCP server (right-click → Stop/Start, or code lenses in `mcp.json`)
- Editing `mcp.json` (VS Code auto-restarts the server)
- `MCP: Restart Server` command
- Server idle timeout (30 min in FinWise) → client reconnects
- `Developer: Reload Window`

Actions that keep the **same MCP session**:
- Opening new chat panels in the same VS Code window
- Multiple Copilot Chat conversations in the same window
- Changing the `Authorization` header (different user token) — same session, different auth

## 12. Comparison with 008.A

| Aspect | 008.A (No Mapping) | 008.B (This Variant) |
|--------|:-------------------:|:-------------------:|
| **New classes** | 1 (`RedisSessionMigrationHandler`) | 3 (`RedisSessionMigrationHandler`, `IMcpSessionMapping`, `RedisMcpSessionMapping`) |
| **Deleted classes** | 1 (`McpSessionMapping`) | 0 (refactored, not deleted) |
| **Redis key families** | 2 (`mcpinit:*`, `agentsession:*`) | 3 (`mcpinit:*`, `mcpsession:*`, `agentsession:*`) |
| **Race conditions to handle** | 0 (no atomic get-or-create mapping) | 1 (`GetOrCreateAgentSessionId` atomicity) |
| **Workflow service changes** | Yes — reset behavior (keep same ID) | No — current reset behavior preserved |
| **Tool code complexity** | 2 lines (get header, call workflow) | 5+ lines (get header, get mapping, call workflow, update mapping) |
| **ID format** | MCP SDK-generated string | Self-generated GUID |
| **Conceptual separation** | MCP transport ↔ agent layer share one ID | MCP transport and agent layer have independent IDs |

**When 008.B is the right choice**: If the project anticipates scenarios where one MCP session could map to multiple agent sessions (e.g., "fork" a conversation), or if the agent session ID format matters to downstream systems, keeping the indirection layer provides flexibility. The current reset behavior (generate new GUID) is preserved without changes to the workflow layer.

**When 008.A is simpler**: If the 1:1 relationship between MCP sessions and agent sessions is permanent, the mapping is pure overhead. 008.A deletes an entire class, removes a Redis key family, eliminates a race condition, and simplifies tool code.

**Honest retrospective**: The mapping layer was never technically necessary — even in the original single-instance, in-memory design. It exists only because the reset logic was written as "generate new GUID" rather than "clear data, keep same key." That early design choice created a problem (MCP session outlives agent session ID), then `McpSessionMapping` was introduced to solve it. 008.B preserves this pattern and externalizes it to Redis, which is a valid approach — but 008.A eliminates the root cause instead of scaling the workaround.
