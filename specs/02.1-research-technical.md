# Technical Research for a Multi-Agent Investment Advisory System

**Research Date**: December 28, 2025  
**Focus**: Technical architecture, design patterns, and technology stack for a .NET-based multi-agent investment advisory system using the Model Context Protocol (MCP)

## Executive Summary

This technical research report synthesizes contemporary best practices, architectural patterns, and technology choices for building a production-grade multi-agent investment advisory system on the .NET platform. The analysis reveals that successful implementations in 2024-2025 leverage several key architectural principles:

1. **MCP as Universal Interface and Integration Layer**: The Model Context Protocol serves a dual architectural role: (a) **FinWise exposes itself AS an MCP server**, enabling end users to access investment advisory capabilities directly from any MCP-compatible AI assistant (Claude Desktop, ChatGPT, GitHub Copilot, etc.) without building custom UIs, and (b) **FinWise agents act AS MCP clients**, consuming external data sources (market data, news, compliance) through standardized MCP server integrations. This dual role makes MCP the universal interface for both human users and external systems.

2. **Multi-Agent Orchestration Patterns**: Modern frameworks support multiple orchestration patterns (Sequential, Concurrent, Handoff, Group Chat, Magentic), each optimized for specific coordination requirements. Investment advisory systems benefit from hybrid patterns combining centralized orchestration for complex routing with decentralized handoffs for specialized analysis.

3. **Tiered Memory Architecture**: Production systems separate short-term conversational context (in-memory with Redis/Valkey for microsecond latency, ADVANCED) from long-term persistent memory (Cosmos DB with vector search for semantic retrieval), achieving both responsiveness and comprehensive historical context.

4. **Event Sourcing for Auditability (ADVANCED/OUT-OF-SCOPE)**: Event sourcing combined with CQRS provides the complete audit trails essential for financial services while enabling sophisticated failure recovery, alternative scenario exploration, and regulatory compliance reporting.

5. **Cost and Performance Optimization (ADVANCED/OUT-OF-SCOPE)**: Context compression techniques, parallel execution patterns, strategic model selection, and prompt caching reduce token costs by 50-95% while maintaining quality through intelligent context management.

The technical stack recommendation centers on:
- **Microsoft Agent Framework** (preview) for agent orchestration and multi-agent patterns (merges Semantic Kernel and AutoGen capabilities)
- **MCP Protocol** for standardized external system integration
- **SQL DB** (Postgress or SQL Server) for persistent storage of limited data such as user's profile and investment recommendations
- **Azure Cosmos DB** for persistent storage with vector search capabilities
- **Redis/Valkey (ADVANCED/OUT-OF-SCOPE)** for high-performance semantic caching
- **Event sourcing with MassTransit or Temporal (ADVANCED/OUT-OF-SCOPE)** for durable workflows
- **OpenTelemetry (ADVANCED/OUT-OF-SCOPE)** for comprehensive observability

## 1. Logical Architecture

### 1.1 Multi-Agent Orchestration Architecture

The foundational architecture for multi-agent investment advisory systems follows an orchestrator-worker pattern where a lead orchestration agent coordinates multiple specialized agents, each responsible for distinct investment domains[2][10]. This pattern directly addresses the core FinWise requirement (FR-1) for "Multi-Agent Workflow with orchestration/triage agent."

The orchestrator agent receives user queries, decomposes complex investment questions into specialized subtasks, routes those subtasks to appropriate specialist agents. For FinWise specifically, the orchestrator would coordinate between:

- **User Profile Agent**: Gathers and maintains investor risk tolerance, investment goals, time horizons, and constraints
- **Global Investment Advisor Agent**: Provides high-level strategic guidance across asset classes (stocks vs. real estate vs. alternatives)
- **Stock Fundamentals Agent** (v0.2): Delivers detailed analysis of individual equities
- **Real Estate Investment Agent** (v0.2): Offers residential real estate investment guidance
- **Investment Strategy Summarization Agent** (v0.2): Synthesizes recommendations into persistent strategy documents
- **Risk Management Agent** (v0.4): Evaluates portfolio composition and risk concentrations

This separation of concerns provides critical benefits for extensibility—new specialized agents can be added without modifying the orchestrator core logic (FR-005)[14][35]. The architecture naturally scales from the baseline v0.1 implementation with two hollow agents to the full vision with eight specialized agents.

### 1.2 MCP Integration Architecture: Dual Role as Server and Client

The Model Context Protocol serves as **both the consumption interface and the integration layer** for FinWise, creating a bidirectional MCP architecture[1][4][18]. This dual role represents a fundamental architectural principle: MCP standardizes how users access FinWise AND how FinWise accesses external systems.

#### 1.2.1 FinWise AS an MCP Server (End-User Consumption Layer)

**FinWise exposes itself as an MCP server**, allowing getting-started investors to access investment advisory capabilities through their preferred AI assistant without requiring a custom UI[1][4][18]. This creates a **zero-friction onboarding experience** where users simply configure FinWise as an MCP server in their existing AI tool.

**End-User Access Pattern**:
```
User → Claude Desktop (MCP Client)
     ↓
     Invokes FinWise MCP Server Tools:
     - assess_risk_profile(userId)
     - get_investment_recommendations(userId, query)
     - analyze_portfolio(userId, holdings)
     ↓
FinWise Orchestrator → Specialized Agents → Investment Recommendations
```

**Supported AI Assistants (MCP Clients)**:
- **Claude Desktop**: Native MCP support, primary target for v0.1
- **ChatGPT**: MCP support via plugins/extensions
- **GitHub Copilot**: MCP integration for developer users
- **Custom Clients**: Any application using MCP client SDKs

**Benefits of MCP Server Exposure**:
- **No custom UI needed for v0.1**: Users access FinWise through familiar AI assistants
- **Cross-platform availability**: Works wherever users have their AI assistant
- **Natural language interface**: Users ask investment questions conversationally
- **Standardized distribution**: Users install FinWise like any other MCP server
- **Rapid iteration**: Backend changes don't require UI updates

This approach directly satisfies the FinWise v0.1 requirement to provide investment advisory capabilities without building a custom frontend. Users interact with FinWise through conversational AI assistants they already use daily.

#### 1.2.2 FinWise AS an MCP Client (External System Integration Layer)

**FinWise agents act as MCP clients**, consuming external data sources through standardized MCP server connections[1][4][18]. This eliminates custom integration code and provides agents with rich contextual data.

**External MCP Servers (3rd-party/industry)**:
- **Market Data Server** (e.g., Alpha Vantage MCP, Yahoo Finance MCP): Real-time quotes, historical pricing, fundamental data
- **Financial News Server** (e.g., Perplexity MCP, NewsAPI MCP): Market intelligence, sentiment analysis, research reports
- **Regulatory Compliance Server** (e.g., SEC EDGAR MCP): Investment rules, constraints, reporting requirements
- **Economic Data Server** (e.g., FRED MCP): Macroeconomic indicators, interest rates, inflation data

**Internal MCP Servers (Custom FinWise Data Access)**:
- **User Profile MCP Server**: Accesses SQL database for user profiles, risk assessments, investment goals
- **Portfolio Management MCP Server**: Accesses SQL database for account data, holdings, transaction history
- **Conversation History MCP Server**: Accesses NoSQL database for chat history, semantic search over past conversations
- **Investment Strategy MCP Server**: Accesses SQL database for prior recommendations, strategy documents, performance tracking

**Agent Access Pattern**:
```
FinWise Agent \u2192 MCP Client Connection \u2192 External MCP Server (Market Data)
            \u2193
            MCP Client Connection \u2192 Internal MCP Server (User Profile DB)
            \u2193
            Synthesize data \u2192 Generate investment recommendations
```

The standardization around MCP eliminates the need for custom integration code. Each server implements the MCP specification, exposing tools (executable functions), resources (data sources), and prompts (reusable templates) through a consistent JSON-RPC API[18]. Agents discover available capabilities dynamically through initialization handshakes, allowing new servers to be added without agent code changes.

**Key Architectural Benefit**: This bidirectional MCP architecture means FinWise is **both consumed via MCP AND integrates via MCP**, creating a consistent protocol for all interactions—whether human users accessing FinWise capabilities or FinWise agents accessing external data.

### 1.3 Agentic Workflow Patterns

The Microsoft Agent Framework provides five primary orchestration patterns, each suited to different coordination requirements[5][14][32]:

**Sequential Orchestration**: Agents execute in a predefined linear sequence, like an assembly line. For FinWise baseline, this pattern works well: User Profile Agent → Global Investment Advisor Agent, where the advisor receives profile context from the profiler.

**Concurrent Orchestration**: Multiple agents work simultaneously on independent tasks, with results aggregated. This pattern optimizes for investment analysis requiring multiple perspectives—asset class analysis, risk assessment, and tax implications can execute in parallel.

**Handoff Orchestration**: Agents dynamically transfer control based on expertise boundaries. When the Global Investment Advisor receives questions about specific stock valuations, it hands off to the Stock Fundamentals Agent (v0.2), preserving conversation context.

**Group Chat Orchestration**: Agents collaborate through multi-turn discussions, critiquing and refining each other's analysis. This pattern suits scenarios where investment recommendations benefit from iterative refinement between analyst and reviewer agents.

**Magentic Orchestration**: A manager agent plans workflow, delegates to specialist agents, tracks progress, and synthesizes responses. This pattern handles complex, multi-phase investment analysis where the optimal sequence isn't predetermined.

For FinWise v0.1, the architecture should support Sequential and Handoff patterns initially, with infrastructure enabling later addition of Concurrent (v0.2) and Magentic (v0.3+) patterns as complexity grows.

## 2. Software Architecture

### 2.1 .NET Technology Stack

**Agent Framework**: Microsoft Agent Framework (preview, released Q4 2025) is the unified successor that merges capabilities from Semantic Kernel and AutoGen[26][28]. The framework provides comprehensive multi-agent orchestration capabilities specifically designed for .NET applications[10][26][28]. Key capabilities include:

- ChatCompletionAgent for LLM-powered conversational agents
- Multiple orchestration patterns through dedicated classes (SequentialOrchestration, ConcurrentOrchestration, HandoffOrchestration, GroupChatOrchestration, MagenticOrchestration)
- Agent memory management with pluggable storage backends
- Kernel plugin system for extending agent capabilities
- InProcessRuntime for managing agent execution lifecycle
- Combines Semantic Kernel's structured orchestration with AutoGen's autonomous agent capabilities

Alternative frameworks considered:
- **Standalone Semantic Kernel or AutoGen**: Both now unified into Microsoft Agent Framework[26][28]
- **LangChain for .NET**: Newer, less mature .NET support compared to Microsoft's frameworks
- **Custom implementation**: Higher development cost, lacks community patterns

**Recommendation**: Use Microsoft Agent Framework for orchestration, with attention to preview API stability. This unified framework provides the best balance of structured orchestration patterns and autonomous agent capabilities.

**MCP Integration**: Microsoft Agent Framework provides native Model Context Protocol support through the official **ModelContextProtocol** NuGet package, maintained collaboratively by Microsoft, Anthropic, and the MCP open protocol organization. The framework enables **both server and client roles**[MCP-SDK][MCP-Docs]:

**MCP Server Capabilities** (FinWise exposes itself to end users):
- Implement MCP server using `ModelContextProtocol.AspNetCore` for HTTP/SSE endpoints
- Expose investment advisory agents as MCP tools via `McpServerTool.Create(agent.AsAIFunction())`
- Support stdio transport for local desktop AI assistants (Claude Desktop, VS Code Copilot)
- Support HTTP/SSE transport for cloud-based AI assistants (ChatGPT, web clients)
- Tool definitions for investment operations (assess_risk_profile, get_recommendations, analyze_portfolio)
- Resource definitions for user data access (profiles, strategies, conversations)

**MCP Client Capabilities** (FinWise agents consume external MCP servers):
- Connect to external MCP servers using `ModelContextProtocol` SDK client APIs (`McpClientFactory.CreateAsync`)
- Three transport options: `MCPStdioTool` (local processes), `MCPStreamableHTTPTool` (remote HTTP), `MCPWebsocketTool` (bidirectional)
- Automatic conversion of MCP tools to Agent Framework `AITool` objects: `mcpTools.Cast<AITool>()`
- Seamless agent integration: `chatClient.CreateAIAgent(tools: [.. mcpTools.Cast<AITool>()])`
- OAuth/bearer token authentication via custom headers for secure server connections
- Works across Agent Framework workflow orchestration patterns (Sequential, Concurrent, Handoff, GroupChat)

**AI Service Integration**: Azure OpenAI Service via the official Azure SDK provides:

- GPT-4 models for complex investment reasoning
- GPT-4o-mini for cost-effective routine tasks
- Structured output support for reliable data extraction
- Token usage tracking for cost management
- Rate limiting and retry policies

### 2.2 Layered Architecture

The system architecture follows a layered approach separating concerns:

**Presentation Layer** (MCP Server Interface - **End-User Consumption**):
- **Exposes FinWise as an MCP server** enabling users to access investment advisory from Claude Desktop, ChatGPT, GitHub Copilot, etc.
- Handles MCP client authentication  and session management (users connecting via their AI assistant)
- Implements MCP protocol exposing:
  - **Tools**: Investment operations (assess_risk_profile, get_investment_recommendations, analyze_portfolio)
  - **Resources**: User data access (user profiles, investment strategies, conversation history)
  - **Prompts**: Guided investment workflows (risk assessment questionnaire, portfolio review template)
- Manages request/response lifecycle for end-user queries from AI assistants
- **Key Benefit**: Zero custom UI needed for v0.1—users access FinWise through familiar AI tools

**Orchestration Layer** (Lead/Triage Agent):
- Implements the triage/routing agent (FR-002)
- Coordinates specialized agents based on query analysis
- Manages conversation state and context passing (FR-004)
- Handles agent handoffs and escalations (FR-016)
- Enforces token budgets and cost controls
- Provides transparency through logging (FR-012)

**Specialized Agent Layer** (Domain Experts):
- User Profile Agent with questionnaire logic and profile storage
- Global Investment Advisor Agent with asset class knowledge
- Future: Stock Fundamentals Agent, Real Estate Agent, etc.
- Each agent operates with curated context and specific tools
- Agents signal completion or escalate to orchestrator

**Integration Layer** (MCP Clients - **External System Integration**):
- **Connects FinWise agents to external MCP servers** for market data, financial news, regulatory compliance
- Manages authentication and connection pooling to external services
- Implements retry and fallback strategies for external data sources
- Provides caching for frequently accessed external data
- **Key Benefit**: Standardized integration without custom connectors for each data source

**Data Layer** (Polyglot Persistence):
- **SQL Database (PostgreSQL/SQL Server)**: Structured, bounded data (user profiles, investment strategies, portfolio holdings, agent configuration)
- **NoSQL Database (Cosmos DB/MongoDB)**: High-growth, semi-structured data (conversations, chat history, RAG embeddings, event sourcing logs, telemetry)
- **Redis/Valkey (ADVANCED/OUT-OF-SCOPE)**: Session state and semantic caching
- **Azure Event Hubs (ADVANCED/OUT-OF-SCOPE)**: Real-time event streaming and audit trail

### 2.2.1 In-process Agent Architecture

**Language Flexibility via MCP Protocol**:

FinWise supports a polyglot architecture where **workflow orchestration** occurs in one language (.NET or Python), but **MCP servers** (tools) can be implemented in any language. This provides maximum flexibility for developer expertise and library availability.

**Architecture Constraints**:

```
✅ SUPPORTED:
┌─────────────────────────────────────┐
│  .NET Workflow Orchestration        │
│  ├─ .NET Agent (Triage)             │
│  ├─ .NET Agent (Profile)            │
│  └─ .NET Agent (Advisor)            │
│       │                              │
│       └─ MCP Tools:                  │
│           ├─ Python MCP Server   ✓  │
│           ├─ .NET MCP Server     ✓  │
│           └─ Node.js MCP Server  ✓  │
└─────────────────────────────────────┘

❌ NOT SUPPORTED:
┌─────────────────────────────────────┐
│  Mixed Language Workflow             │
│  ├─ .NET Agent (Triage)             │
│  ├─ Python Agent (Profile)      ✗   │
│  └─ .NET Agent (Advisor)            │
└─────────────────────────────────────┘
Cannot mix Python and .NET agents
in same workflow runtime
```

**Recommended Implementation Pattern**:

1. **Choose Primary Language for Workflow Orchestration**:
   - **Option A (.NET Primary)**: Implement triage agent, orchestration, and core agents in .NET
   - **Option B (Python Primary)**: Implement triage agent, orchestration, and core agents in Python

2. **Implement MCP Servers in Optimal Language**:
   - **Python MCP Servers**: Market data (leverages `yfinance`, `pandas`), ML models (leverages `scikit-learn`, `tensorflow`)
   - **.NET MCP Servers**: Azure integrations (leverages Azure SDK), enterprise compliance (leverages .NET libraries)
   - **Mixed Approach**: Use both Python and .NET MCP servers based on task requirements

3. **Protocol-Level Interoperability**:
   - MCP protocol is JSON-RPC over stdio/HTTP/WebSocket — language-neutral
   - .NET workflow can consume Python MCP servers seamlessly
   - Python workflow can consume .NET MCP servers seamlessly

**FinWise v0.1 Recommendation**:
- **Workflow Orchestration**: .NET (Microsoft Agent Framework for .NET)
- **Core Agents**: .NET (Triage, Profile, Advisor agents)
- **MCP Servers**: Python for market data (leverages `yfinance`), .NET for compliance/Azure integration
- **Rationale**: .NET provides strong Azure integration, type safety, and performance for core orchestration; Python excels at data science and has mature financial libraries for MCP tools

### 2.2.2 Distributed and Remote Agent Architecture via A2A Protocol (v0.4+)

**Advanced Distributed Agent Architecture**:

For v0.4+ deployments, FinWise can adopt a **distributed agent architecture** using the **A2A (Agent-to-Agent) protocol**. This enables agents to run as **independent HTTP services** that communicate over a standardized protocol, supporting:

- **Polyglot Remote Agents**: Some agents written in .NET, others in Python, deployed on different servers
- **Independent Scaling**: Scale individual agents based on workload (e.g., scale Market Analyst agent separately from Compliance agent)
- **Language-Neutral Communication**: Agents communicate via JSON-RPC over HTTP, regardless of implementation language
- **Service Resilience**: Agent failures are isolated; system continues operating with reduced capabilities

**A2A Architecture Pattern**:

```
┌─────────────────────────────────────────────────────────────┐
│  .NET Orchestrator Service (Port 5000)                      │
│  ├─ Triage Agent (local .NET agent)                         │
│  └─ Connects to remote agents via A2A protocol:             │
└─────────────────────────────────────────────────────────────┘
         │                          │                          │
         ▼                          ▼                          ▼
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│ Python Service   │  │ .NET Service     │  │ Python Service   │
│ (Port 5001)      │  │ (Port 5002)      │  │ (Port 5003)      │
│                  │  │                  │  │                  │
│ Market Analyst   │  │ Compliance Agent │  │ Portfolio Mgr    │
│ - Uses yfinance  │  │ - Azure SDK      │  │ - pandas/scipy   │
│ - pandas/numpy   │  │ - Regex patterns │  │ - optimization   │
└──────────────────┘  └──────────────────┘  └──────────────────┘

A2A Protocol: JSON-RPC over HTTP
Agent Discovery: /.well-known/agent-card.json
```

**Benefits for FinWise v0.4+**:
1. **Independent Deployment**: Update Market Analyst agent without redeploying core orchestrator
2. **Team Autonomy**: Python team owns Market Analyst, .NET team owns Compliance agent
3. **Polyglot Optimization**: Use best language for each agent's domain (Python for data science, .NET for Azure)
4. **Horizontal Scaling**: Deploy multiple instances of high-demand agents behind load balancer
5. **Service Isolation**: Market data service outage doesn't crash entire system

**Implementation Example (.NET Orchestrator + Remote Agents)**:

```csharp
// .NET Orchestrator connects to remote Python and .NET agents via A2A
using Microsoft.Agents.AI.A2A;
using A2A;

// Connect to remote Python market analyst agent (HTTP endpoint)
var marketAnalystUri = new Uri("https://market-analyst.finwise.com/a2a/");
AIAgent remoteMarketAnalyst = new A2AClient(marketAnalystUri).GetAIAgent();

// Connect to remote .NET compliance agent
var complianceUri = new Uri("https://compliance.finwise.com/a2a/");
AIAgent remoteComplianceAgent = new A2AClient(complianceUri).GetAIAgent();

// Local triage agent uses remote agents as tools via A2A protocol
var tools = new List<AITool>
{
    (AITool)remoteMarketAnalyst.AsAIFunction(),  // Python agent as tool
    (AITool)remoteComplianceAgent.AsAIFunction()  // .NET agent as tool
};

ChatClientAgent triageAgent = new(chatClient,
    "You coordinate specialist agents to answer investment questions.",
    "triage_agent",
    tools: tools);

// When triage agent calls remoteMarketAnalyst tool, it sends A2A HTTP request
var response = await triageAgent.RunAsync("What are the top tech stocks to buy?");
// A2A protocol: { "method": "message/send", "params": { "message": {...} } }
```

**Python Remote Agent Implementation (A2A Server)**:

```python
# market_analyst_agent.py - Python agent exposed via A2A protocol
from agent_framework import ChatAgent
from agent_framework.openai import OpenAIChatClient
from fastapi import FastAPI
import yfinance as yf

# Define market analysis agent
agent = ChatAgent(
    chat_client=OpenAIChatClient(),
    name="MarketAnalyst",
    instructions="You provide market analysis using real-time stock data.",
    tools=[get_stock_quote, analyze_trends]  # Custom Python functions
)

# Expose agent via A2A protocol HTTP endpoint
app = FastAPI()

@app.post("/a2a/")
async def handle_a2a_request(request: Request):
    """A2A protocol endpoint - receives JSON-RPC requests from .NET orchestrator"""
    body = await request.json()
    
    # A2A protocol: extract message from JSON-RPC
    message = body["params"]["message"]["parts"][0]["text"]
    
    # Run agent
    result = await agent.run(message)
    
    # Return A2A protocol response (JSON-RPC)
    return {
        "id": body["id"],
        "jsonrpc": "2.0",
        "result": {
            "message": {
                "kind": "message",
                "role": "assistant",
                "parts": [{"kind": "text", "text": str(result)}]
            }
        }
    }

# Run: uvicorn market_analyst_agent:app --host 0.0.0.0 --port 5001
```

### 2.3 Communication Patterns

**Agent-to-Agent Communication (In-Process)**:
Agents communicate through handoff orchestration where a triage agent dynamically routes to specialist agents based on query context[10]:

```csharp
using Microsoft.Agents.AI;
using Microsoft.Agents.AI.Workflows;

// v0.1-v0.3: In-process agents within same application
ChatClientAgent profileAgent = new(chatClient,
    "You assess investor risk tolerance, goals, and constraints. Only handle profile-related queries.",
    "profile_agent",
    "Specialist for user profile and risk assessment");

ChatClientAgent investmentAdvisor = new(chatClient,
    "You provide investment recommendations across asset classes. Only handle investment advice.",
    "investment_advisor",
    "Specialist for investment strategy and recommendations");

ChatClientAgent triageAgent = new(chatClient,
    "You analyze user queries and route to appropriate specialist agents. ALWAYS handoff to another agent.",
    "triage_agent",
    "Routes queries to profile or investment specialists");

// Build handoff workflow with bidirectional routing
Workflow workflow = AgentWorkflowBuilder.CreateHandoffBuilderWith(triageAgent)
    .WithHandoffs(triageAgent, [profileAgent, investmentAdvisor])
    .WithHandoffs([profileAgent, investmentAdvisor], triageAgent)
    .Build();

// Execute workflow with streaming
List<ChatMessage> messages = [new(ChatRole.User, "What should I invest in given my conservative risk profile?")];
await using StreamingRun run = await InProcessExecution.StreamAsync(workflow, messages);
await run.TrySendMessageAsync(new TurnToken(emitEvents: true));

// Process workflow events
await foreach (WorkflowEvent evt in run.WatchStreamAsync())
{
    if (evt is WorkflowOutputEvent output)
    {
        var outputMessages = output.As<List<ChatMessage>>();
        Console.WriteLine(outputMessages?[^1].Text);
    }
}
```

**Agent-to-MCP Server Communication (Polyglot Support)**:

Agents consume MCP server tools through direct Agent Framework integration. **MCP servers can be implemented in Python, .NET, or any language** — the protocol is language-neutral (JSON-RPC over stdio/HTTP/WebSocket). MCP tools are automatically converted to AITool objects:

```csharp
using ModelContextProtocol.Client;
using ModelContextProtocol.Protocol;
using Microsoft.Agents.AI;

// Example 1: Connect to Python MCP server via stdio
await using var pythonMcpClient = await McpClientFactory.CreateAsync(
    new StdioClientTransport(new()
    {
        Name = "MarketDataServer",
        Command = "python",
        Arguments = ["market_data_server.py"]  // Python MCP server
    })
);

// Example 2: Connect to .NET MCP server via stdio
await using var dotnetMcpClient = await McpClientFactory.CreateAsync(
    new StdioClientTransport(new()
    {
        Name = "ComplianceServer",
        Command = "dotnet",
        Arguments = ["run", "--project", "compliance_server.csproj"]  // .NET MCP server
    })
);

// Retrieve tools from both servers
var pythonTools = await pythonMcpClient.ListToolsAsync();
var dotnetTools = await dotnetMcpClient.ListToolsAsync();

// Create agent with tools from multiple polyglot MCP servers
var agent = chatClient.CreateAIAgent(
    name: "MarketAnalyst",
    instructions: "You analyze market data and ensure regulatory compliance",
    tools: [
```

- ✅ **Workflow orchestration**: Can be .NET OR Python (Microsoft Agent Framework supports both)

- ✅ **Recommended pattern**:

  - **v0.4+**: Use A2A protocol to deploy agents in different languages as independent HTTP services

## 3. Data Architecture

### 3.1 Multi-Tier Memory Architecture

Production multi-agent systems require sophisticated memory management balancing performance, cost, and completeness[3][6][20]. The architecture follows Google's three-tier approach:

**Tier 1: Working Context** (Active Agent Memory) - **v0.1 (OUT-OF-THE-BOX)**
Microsoft Agent Framework provides automatic conversation history management through `AgentThread`:
- ✅ **Automatic in-memory conversation storage**: `InMemoryChatMessageStore` accumulates all messages (user + assistant)
- ✅ **Last 5-10+ conversational turns maintained**: Full conversation history stored by default
- ✅ **Current task context and intermediate results**: All messages automatically captured
- ✅ **Zero configuration required**: Framework manages lifecycle via `AgentThread.GetNewThread()`
- ✅ **Automatic context passing**: Agent receives complete conversation history on each invocation
- **Lifespan**: Single session (lost on application restart without persistence)

**What Requires Custom Implementation (v0.4)**:
- ❌ **Thread persistence across sessions**: Must serialize `AgentThread` and store in database
- ❌ **Chat history size management**: Must configure `IChatReducer` with message/token limits
- ❌ **Semantic memory extraction**: Must implement custom `AIContextProvider` to extract facts (user preferences, risk tolerance)

```csharp
// v0.1: Out-of-the-box conversation history
AIAgent agent = chatClient.CreateAIAgent(instructions: "You are a helpful assistant.");
AgentThread thread = agent.GetNewThread();  // Automatic history management
await agent.RunAsync("What's my risk tolerance?", thread);
await agent.RunAsync("Based on that, what should I invest in?", thread);  // Agent remembers previous turn

// v0.4: Custom history reduction (optional optimization)
AIAgent optimizedAgent = chatClient.CreateAIAgent(new ChatClientAgentOptions
{
    Instructions = "You are a helpful assistant.",
    ChatMessageStoreFactory = ctx => new InMemoryChatMessageStore(
        new MessageCountingChatReducer(10),  // Keep last 10 messages only
        ctx.SerializedState, ctx.JsonSerializerOptions,
        InMemoryChatMessageStore.ChatReducerTriggerEvent.AfterMessageAdded)
});
```

**Tier 2: Session Memory** (Redis/Valkey Semantic Cache) - **(ADVANCED/OUT-OF-SCOPE)**
- High-performance cache of recent session state
- Semantic vector search over conversation history
- User preferences and profile summary
- Recent investment analysis results
- Managed by ElastiCache for Valkey with RediSearch[22][23]
- Lifespan: User session (hours to days)

**Tier 3: Long-Term Memory** (Polyglot Persistent Storage) - **v0.1 baseline (SQL), v0.2+ (NoSQL expansion)**
- **SQL Database (PostgreSQL with pgvector or SQL Server)**: User profiles, investment strategies, portfolio holdings, transactional data with ACID guarantees
- **NoSQL Database (Cosmos DB/MongoDB)**: Conversation history with embeddings, chat messages, RAG document store, event sourcing logs
- Polyglot approach optimizes cost (SQL for small datasets) and performance (NoSQL for high-growth data)
- Lifespan: Permanent (years)

**Memory Flow** (v0.1 baseline → v0.4 advanced):
1. **v0.1**: Agent receives user query → Working context loaded automatically from `AgentThread`
2. **v0.2+**: Agent performs analysis → Optionally writes to session cache (Redis - ADVANCED/OUT-OF-SCOPE)
3. **v0.1**: Agent completes task → Persists results to SQL database (profiles, strategies)
4. **v0.4**: Session idle timeout → Serializes and stores `AgentThread` for cross-session persistence
5. **v0.4**: Next session start → Deserializes stored thread to resume conversation

**Persistence Implementation (v0.4 - Custom)**:
```csharp
// Serialize thread for cross-session persistence
JsonElement serializedThread = thread.Serialize();
await database.SaveConversationAsync(userId, serializedThread);

// Resume conversation in future session
var storedThread = await database.LoadConversationAsync(userId);
AgentThread resumedThread = agent.DeserializeThread(storedThread);
await agent.RunAsync("Continue our previous conversation", resumedThread);
```

**Semantic Memory Extraction (v0.4 - Custom AIContextProvider)**:
```csharp
// Extract and inject long-term memories (user preferences, risk tolerance)
public class InvestorProfileMemory : AIContextProvider
{
    public override async ValueTask InvokedAsync(InvokedContext context, CancellationToken ct)

```

This tiered approach achieves microsecond retrieval of active context while maintaining complete historical records for compliance.

### 3.2 Vector Search for Semantic Memory

ADRIAN's note: Sessions storage in DB (Vector Search DB) is advanced, but we'll keep it in the scope for v0.4, so the system/app can have access to older sessions/onversations from the users.

Vector embeddings enable semantic search across investment conversations and knowledge[8][43][46]. When a user asks "How did you recommend handling the 2023 market volatility?", vector search retrieves contextually similar prior analyses even if different keywords were used.

**Embedding Strategy**:
- Generate embeddings for each conversational turn, investment recommendation, and strategy document
- Use Azure OpenAI text-embedding-ada-002 for consistent embedding space
- Store conversation embeddings in NoSQL database (Cosmos DB/MongoDB) for high-volume vector search
- Store strategy embeddings in SQL database (PostgreSQL with pgvector) for relational integrity

**Semantic Query Pattern**:
```csharp

    .GetContainer("FinWise", "Conversations")
    .VectorSearchAsync(queryEmbedding, limit: 5);

// Agent receives semantically relevant prior discussions
foreach (var conversation in similarConversations)
{
    context.Append($"Prior analysis: {conversation.Summary}");
}
```

**Index Configuration**:
- Cosmos DB vector index with 1536 dimensions (ada-002 dimensionality)
- Cosine similarity for relevance scoring
- Hybrid search combining vector similarity with metadata filters (user, date range, asset class)

- **Total per user**: Potentially GBs over time (exponential growth)

### 3.4 Event Sourcing Schema (ADVANCED / OUT-OF-SCOPE)

For audit trails and regulatory compliance, event sourcing captures all state changes as immutable events[9][12][55]:

**Event Store (Stored in NoSQL Database)**:

Event sourcing logs are stored in the NoSQL database (EventSourcingEvents collection shown above) where append-only writes and high-volume scalability are required.

**Event Types for Investment Advisory**:
- `UserProfileCreated`: New user onboarding
- `UserProfileUpdated`: Risk tolerance or goal changes
- `InvestmentAnalysisRequested`: User initiates portfolio analysis
- `MarketDataRetrieved`: Agent fetches external market data
- `RecommendationGenerated`: Agent produces investment recommendation
- `StrategyApproved`: User approves investment strategy
- `StrategyRejected`: User declines recommendation
- `AgentHandoff`: Orchestrator transfers between agents

## 4. Data Flow Patterns

### 4.1 Request Flow: User Query to Investment Recommendation

**Phase 1: Query Reception and Session Initialization**

```
User → MCP Client (Claude/ChatGPT/Copilot) 
    → FinWise MCP Server → Orchestrator Agent
```

1. User submits query through MCP-compatible client
2. MCP client sends JSON-RPC request to FinWise MCP server
3. FinWise MCP server validates authentication and session
4. Request routed to Orchestrator Agent with session context

**Phase 2: Context Loading and Agent Routing**

```
Orchestrator Agent → Session Cache (Redis - ADVANCED/OUT-OF-SCOPE) → Long-term Memory (SQL + NoSQL)
    ↓
Agent Routing Decision
    ↓
Specialized Agent Invocation
```

1. Orchestrator loads session context from Redis cache (ADVANCED/OUT-OF-SCOPE)
2. If cache fails (ADVANCED/OUT-OF-SCOPE), hydrate from persistent storage (IN SCOPE: Always read from persistent storage database):
   - User profile from SQL database (structured, fast lookup) (v0.1)
   - Recent conversation history from NoSQL database (semantic search)  (v0.4)
3. Orchestrator analyzes query to determine routing:
   - "What's my risk tolerance?" → Profile Agent only
   - "What investments match my profile?" → Profile Agent → Advisor Agent (sequential)
   - "Should I buy MSFT?" → Advisor Agent → escalates to orchestrator (future: Stock Fundamentals Agent)
4. Orchestrator invokes appropriate agent(s) with curated context

**Phase 3: Specialized Agent Processing**

```
Specialized Agent → MCP Servers (Market Data, Compliance, etc.)
    ↓
Agent Reasoning
    ↓
Result Generation
```

1. Agent receives task and context from orchestrator
2. Agent invokes MCP server tools as needed:
   - Market data server for current quotes
   - News server for sentiment analysis
   - Compliance server for regulatory checks
3. Agent performs reasoning using LLM with tool results
4. Agent generates response or signals completion

**Phase 4: Response Aggregation and Persistence**

```
Specialized Agent(s) → Orchestrator Agent
    ↓
Response Synthesis (By Summarization Agent)
    ↓
Session Cache (ADVANCED/OUT-OF-SCOPE) + Long-term Storage (SQL + NoSQL) + Event Log
    ↓
MCP Server → MCP Client → User
```

1. Specialized agent(s) return results to orchestrator
2. Do synthesis of multi-agent findings (by Summarization agent, v0.2)
3. Results persisted to:
   - Session cache (Redis) for immediate access  (ADVANCED/OUT-OF-SCOPE)
   - SQL database (user profiles, investment strategies) for structured data
   - NoSQL database (conversation messages with embeddings) for chat history
   - NoSQL event log (EventSourcingEvents collection) for audit trail
4. Final response returns through MCP protocol to user

### 4.2 Context Passing Between Agents

Context passing determines how effectively agents leverage each other's work. The Microsoft Agent Framework provides explicit mechanisms:

**Sequential Context Passing**:

4. Orchestrator invokes Advisor Agent with accumulated history

## 5. Design Patterns

### 5.1 Orchestration Patterns

**Pattern 1: Triage Router Pattern (Handoff Workflow)**

The orchestrator analyzes user queries and routes to appropriate specialized agents using Microsoft Agent Framework's handoff orchestration[32][35]:

```csharp
using Microsoft.Agents.AI;
using Microsoft.Agents.AI.Workflows;
using Azure.AI.OpenAI;
using Azure.Identity;

// Create specialist agents with focused responsibilities
var endpoint = new Uri(Environment.GetEnvironmentVariable("AZURE_OPENAI_ENDPOINT")!);
var chatClient = new AzureOpenAIClient(endpoint, new AzureCliCredential())
    .GetChatClient("gpt-4o-mini");

ChatClientAgent profileAgent = new(chatClient,
    "You assess investor risk tolerance, goals, and constraints. Only handle profile-related queries.",
    "profile_agent",
    "Specialist for user profile and risk assessment");

ChatClientAgent investmentAdvisor = new(chatClient,
    "You provide investment recommendations across asset classes. Only handle investment advice.",
    "investment_advisor",
    "Specialist for investment strategy and recommendations");

ChatClientAgent triageAgent = new(chatClient,
    "You analyze user queries and determine which specialist agent to use. ALWAYS handoff to another agent.",
    "triage_agent",
    "Routes queries to profile or investment specialists");

// Build handoff workflow with bidirectional routing
Workflow workflow = AgentWorkflowBuilder.CreateHandoffBuilderWith(triageAgent)
    .WithHandoffs(triageAgent, [profileAgent, investmentAdvisor])
    .WithHandoffs([profileAgent, investmentAdvisor], triageAgent)
    .Build();

// Execute workflow with streaming
public async Task<string> RouteAndProcessAsync(string userQuery)
{
    List<ChatMessage> messages = [new(ChatRole.User, userQuery)];
    await using StreamingRun run = await InProcessExecution.StreamAsync(workflow, messages);
    await run.TrySendMessageAsync(new TurnToken(emitEvents: true));
    
    string? lastAgentResponse = null;
    
    // Process workflow events to get final response
    await foreach (WorkflowEvent evt in run.WatchStreamAsync())
    {
        if (evt is WorkflowOutputEvent output)
        {
            var outputMessages = output.As<List<ChatMessage>>();
            if (outputMessages?.Count > 0)
            {
                lastAgentResponse = outputMessages[^1].Text;
            }
        }
    }
    
    return lastAgentResponse ?? "I can help with investment questions and profile management.";
}
```

**Pattern 2: Agent Capability Registry**

Agents register capabilities dynamically, enabling orchestrator to route without hardcoding[14]:

```csharp
public interface ISpecializedAgent
{
    string AgentName { get; }
    string Description { get; }
    IEnumerable<string> Capabilities { get; }
    Task<bool> CanHandleAsync(string query);
    Task<ChatMessageContent> ProcessAsync(string query, ChatHistory context);
}

public class AgentRegistry
{
    private readonly Dictionary<string, ISpecializedAgent> _agents = new();

    public void RegisterAgent(ISpecializedAgent agent)
    {
        _agents[agent.AgentName] = agent;
    }

    public async Task<ISpecializedAgent> FindAgentForQueryAsync(string query)
    {
        foreach (var agent in _agents.Values)
        {
            if (await agent.CanHandleAsync(query))
            {
                return agent;
            }
        }
        return null; // Fallback to default agent
    }
}
```

```csharp
public class ContextCurator
{
    public ChatHistory CurateForAgent(
        ISpecializedAgent agent, 
        ChatHistory fullHistory, 
        string currentQuery)
    {
        var curatedHistory = new ChatHistory();

        // Add system message defining agent role
        curatedHistory.AddSystemMessage(agent.Description);

        // Extract relevant prior context using semantic search
        var relevantMessages = SemanticSearch(fullHistory, currentQuery, limit: 5);
        foreach (var msg in relevantMessages)
        {
            curatedHistory.AddMessage(msg);
        }

        // Add current query
        curatedHistory.AddUserMessage(currentQuery);

        return curatedHistory;
    }

    private IEnumerable<ChatMessageContent> SemanticSearch(
        ChatHistory history, 
        string query, 
        int limit)
    {
        // Generate query embedding
        var queryEmbedding = _embeddingService.GenerateEmbedding(query);

        // Find most similar messages
        return history
            .Select(msg => new 
            {
                Message = msg,
                Similarity = CosineSimilarity(queryEmbedding, msg.Embedding)
            })
            .OrderByDescending(x => x.Similarity)
            .Take(limit)
            .Select(x => x.Message);
    }
}
```

```csharp
public class SessionStateManager
{
    private readonly IDistributedCache _cache;  // Redis  (ADVANCED)
    private readonly CosmosClient _cosmosClient;  // Cosmos DB

    public async Task<SessionState> LoadSessionAsync(string sessionId)
    {
        // Try cache first
        var cachedState = await _cache.GetStringAsync($"session:{sessionId}");
        if (cachedState != null)
        {
            return JsonSerializer.Deserialize<SessionState>(cachedState);
        }

        // Cache miss - hydrate from Cosmos DB
        var container = _cosmosClient.GetContainer("FinWise", "Sessions");
        var response = await container.ReadItemAsync<SessionState>(
            sessionId, 
            new PartitionKey(sessionId));

        // Populate cache for next access
        await _cache.SetStringAsync(
            $"session:{sessionId}", 
            JsonSerializer.Serialize(response.Resource),
            new DistributedCacheEntryOptions 
            { 
                AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(24) 
            });

        return response.Resource;
    }

    public async Task SaveSessionAsync(SessionState state)
    {
        // Update cache immediately
        await _cache.SetStringAsync(
            $"session:{state.SessionId}",
            JsonSerializer.Serialize(state),
            new DistributedCacheEntryOptions 
            { 
                AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(24) 
            });

        // Async persist to Cosmos DB
        _ = Task.Run(async () =>
        {
            var container = _cosmosClient.GetContainer("FinWise", "Sessions");
            await container.UpsertItemAsync(state, new PartitionKey(state.SessionId));
        });
    }
}
```

**Pattern 2: Agent Memory Pattern**

Agents maintain separate short-term (working) and long-term (persistent) memory[3][6][20]:

```csharp
public class AgentMemory
{
    private readonly IMemoryStore _shortTermMemory;  // In-memory or Redis  (ADVANCED / OUT-OF-SCOPE)
    private readonly IMemoryStore _longTermMemory;   // Cosmos DB

    public async Task RememberAsync(string key, string value, MemoryType type)
    {
        if (type == MemoryType.ShortTerm)
        {
            await _shortTermMemory.SetAsync(key, value, TimeSpan.FromHours(1));
        }
        else
        {
            await _longTermMemory.SetAsync(key, value, TimeSpan.MaxValue);
        }
    }

    public async Task<string> RecallAsync(string key)
    {
        // Check short-term first
        var shortTermValue = await _shortTermMemory.GetAsync(key);
        if (shortTermValue != null)
        {
            return shortTermValue;
        }

        // Fall back to long-term
        var longTermValue = await _longTermMemory.GetAsync(key);
        if (longTermValue != null)
        {
            // Promote to short-term for faster future access
            await _shortTermMemory.SetAsync(key, longTermValue, TimeSpan.FromHours(1));
        }

        return longTermValue;
    }
}
```

### 5.3 Error Handling and Resilience Patterns

**Pattern 1: Circuit Breaker for External Services (ADVANCED / OUT-OF-SCOPE)**

Prevent cascading failures when MCP servers become unavailable[48]:

```csharp
public class ResilientMcpClient
{
    private readonly McpClient _innerClient;
    private readonly IAsyncPolicy _circuitBreakerPolicy;

    public ResilientMcpClient(McpClient innerClient)
    {
        _innerClient = innerClient;
        _circuitBreakerPolicy = Policy
            .Handle<McpException>()
            .CircuitBreakerAsync(
                handledEventsAllowedBeforeBreaking: 3,
                durationOfBreak: TimeSpan.FromMinutes(1),
                onBreak: (exception, duration) =>
                {
                    _logger.LogWarning($"Circuit breaker opened for {duration}");
                },
                onReset: () =>
                {
                    _logger.LogInformation("Circuit breaker reset");
                });
    }

    public async Task<ToolCallResult> CallToolAsync(CallToolRequest request)
    {
        return await _circuitBreakerPolicy.ExecuteAsync(async () =>
        {
            return await _innerClient.CallToolAsync(request);
        });
    }
}
```

**Pattern 2: Fallback Strategy (ADVANCED / OUT-OF-SCOPE)**

Gracefully degrade when preferred services fail:

```csharp
public class FallbackMarketDataService
{
    private readonly IMcpClient _primaryClient;
    private readonly IMcpClient _secondaryClient;
    private readonly ICache _cache;

    public async Task<MarketQuote> GetQuoteAsync(string symbol)
    {
        try
        {
            // Try primary market data server
            return await _primaryClient.CallToolAsync("GetQuote", new { symbol });
        }
        catch (McpException)
        {
            _logger.LogWarning("Primary market data failed, trying secondary");
            try
            {
                // Fallback to secondary server
                return await _secondaryClient.CallToolAsync("GetQuote", new { symbol });
            }
            catch (McpException)
            {
                _logger.LogError("Secondary market data failed, using cache");
                // Last resort: return cached data
                var cached = await _cache.GetAsync<MarketQuote>($"quote:{symbol}");
                if (cached != null)
                {
                    _logger.LogWarning($"Returning stale quote for {symbol}");
                    return cached with { IsStale = true };
                }

                throw new InvalidOperationException(
                    "Market data unavailable and no cached data exists");
            }
        }
    }
}
```

```python
# market_data_server.py - Python MCP server consumed by .NET workflow
import asyncio
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool
import yfinance as yf

server = Server("market-data-mcp")

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="get_stock_quote",
            description="Get current stock quote for a symbol",
            inputSchema={
                "type": "object",
                "properties": {
                    "symbol": {"type": "string", "description": "Stock ticker symbol"}
                },
                "required": ["symbol"]
            }
        )
    ]

@server.call_tool()
async def call_tool(name: str, arguments: dict) -> str:
    if name == "get_stock_quote":
        ticker = yf.Ticker(arguments["symbol"])
        price = ticker.history(period="1d")["Close"].iloc[-1]
        return f"Current price of {arguments['symbol']}: ${price:.2f}"
    return "Unknown tool"

async def main():
    async with stdio_server() as (read_stream, write_stream):
        await server.run(read_stream, write_stream)

if __name__ == "__main__":
    asyncio.run(main())
```

```csharp
public class TokenBudgetGuard
{
    private readonly int _maxTokensPerRequest;
    private readonly ITokenCounter _tokenCounter;

    public async Task<ChatCompletion> ExecuteWithBudgetAsync(
        Func<Task<ChatCompletion>> operation,
        ChatHistory history)
    {
        var inputTokens = _tokenCounter.CountTokens(history);
        if (inputTokens > _maxTokensPerRequest)
        {
            throw new InvalidOperationException(
                $"Request exceeds token budget: {inputTokens} > {_maxTokensPerRequest}");
        }

        var result = await operation();

        var totalTokens = result.Usage.InputTokens + result.Usage.OutputTokens;
        if (totalTokens > _maxTokensPerRequest)
        {
            _logger.LogWarning(
                $"Response exceeded budget: {totalTokens} tokens used");
        }

        return result;
    }
}
```

```csharp
// orchestrator/HealthCheck.cs
public class A2AHealthCheck : IHealthCheck
{
    private readonly IEnumerable<Uri> _agentUris;
    
    public async Task<HealthCheckResult> CheckHealthAsync(HealthCheckContext context)
    {
        var results = new Dictionary<string, object>();
        
        foreach (var uri in _agentUris)
        {
            try
            {
                var cardUri = new Uri(uri, "/.well-known/agent-card.json");
                var card = await httpClient.GetFromJsonAsync<AgentCard>(cardUri);
                results[uri.Host] = "healthy";
            }
            catch
            {
                results[uri.Host] = "unhealthy";
            }
        }
        
        return results.Values.All(v => v.ToString() == "healthy")
            ? HealthCheckResult.Healthy()
            : HealthCheckResult.Degraded("Some agents unavailable", data: results);
    }
}
```

**Benefits of A2A Architecture (v0.4+)**:
- **Polyglot Agents**: Market analyst in Python (yfinance), compliance in .NET (Azure SDK)
- **Independent Deployment**: Update market analyst without redeploying orchestrator
- **Horizontal Scaling**: Scale high-demand agents independently
- **Service Isolation**: Compliance agent failure doesn't crash market analyst
- **Team Autonomy**: Python team owns market analyst, .NET team owns compliance
- **HTTP Discovery**: Agents publish capabilities via agent cards (/.well-known/agent-card.json)
- **Protocol Standardization**: A2A protocol enables interoperability across frameworks

**When to Choose A2A (v0.4+) vs In-Process (v0.1-v0.3)**:

| Factor | In-Process (v0.1-v0.3) | A2A Distributed (v0.4+) |
|--------|------------------------|--------------------------|
| **Deployment Complexity** | Low (single service) | High (multiple services) |
| **Latency** | Lowest (in-memory) | Higher (HTTP overhead) |
| **Polyglot Support** | No (agents must be same language) | Yes (Python + .NET agents) |
| **Scalability** | Vertical (scale entire app) | Horizontal (scale individual agents) |
| **Fault Isolation** | No (agent crash = app crash) | Yes (agent crash isolated) |
| **Team Autonomy** | Low (shared codebase) | High (separate repos/teams) |
| **Development Velocity** | Fast (no HTTP setup) | Slower (manage services) |

## Conclusion

This technical research establishes a comprehensive foundation for implementing FinWise as a production-grade multi-agent investment advisory system. The recommended architecture leverages:

1. **Microsoft Agent Framework** (preview) for structured multi-agent orchestration with proven patterns (Sequential, Handoff, Concurrent) — unified framework combining Semantic Kernel's orchestration with AutoGen's autonomous capabilities
2. **Model Context Protocol (MCP)** for standardized, extensible integration with external data sources and specialized tools

The architecture is designed for incremental implementation, supporting the v0.1 baseline with two hollow agents while providing clear paths to v0.2 (specialized agents with external data) and v0.3+ (custom UI, advanced orchestration). Each architectural decision balances immediate implementation simplicity with long-term extensibility, ensuring the system can evolve from proof-of-concept to production-grade financial advisory platform.

The technical stack prioritizes Azure-native services for enterprise security and scalability, .NET 10 for performance and modern language features, and open standards (MCP, OpenTelemetry) for flexibility and vendor independence.
