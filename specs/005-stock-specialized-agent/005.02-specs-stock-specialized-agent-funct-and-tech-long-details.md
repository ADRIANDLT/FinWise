## Functional Specification

### Problem Statement

The FinWise multi-agent workflow currently has two agents: **UserProfileAgent** (profile collection) and **AdvisorAgent** (personalized investment advice). The AdvisorAgent provides general investment guidance but has no access to real company financial data — it cannot answer questions about specific stock fundamentals, annual report details, or company financials.

An Azure Foundry agent (`stock-specialized-investment-agent`) already exists and is **pre-provisioned** in Azure AI Foundry, grounded on actual stock documents (Apple, Microsoft, Tesla, Nvidia, and Amazon annual reports). This agent needs to be integrated into the existing FinWise multi-agent workflow as a new spoke agent so users can ask stock-specific questions and get answers backed by real financial documents.

### Proposed Solution

Add a **StockSpecializedAgent** as a fourth agent (third spoke) in the existing hub-and-spoke multi-agent workflow. Unlike the other agents which are `ChatClientAgent` instances created locally with a local prompt and the Azure OpenAI `IChatClient`, this agent is the pre-provisioned **Azure Foundry native `AIAgent`** — its LLM model, prompt/instructions, and file search grounding are all defined and managed in Azure AI Foundry, not in local code. It is resolved once at startup via `AIProjectClient.GetAIAgentAsync()` and placed directly into the `AgentWorkflow` handoff topology as a peer of the other spokes. The `Microsoft.Agents.AI.AzureAI` bridge package enables this Foundry agent to participate seamlessly in the Microsoft Agent Framework's handoff mechanism.

**User interaction flow:**
1. User completes profile (existing flow, unchanged)
2. User asks a stock-specific question (e.g., "What was Apple's revenue in 2024?")
3. Orchestrator hands off directly to the Foundry-hosted stock agent (no local intermediary)
4. The Foundry agent searches its grounded stock documents and returns an answer
5. Response flows back through the workflow to the user

**No new MCP server. No new CLI.** The agent is added directly into `FinWise.MultiAgentWorkflow` as a new spoke in the existing handoff topology.

### Functional Requirements

- [x] FR-1 — Users can ask stock-specific questions (company financials, annual reports, revenue, earnings) and receive answers grounded in real financial documents
- [x] FR-2 — The Orchestrator routes stock/company-specific queries to the StockSpecializedAgent and personalized advice queries to the AdvisorAgent. Routing is broad: anything touching stocks at all goes to the stock agent.
- [x] FR-2b — The AdvisorAgent hands off stock-specific questions to the **orchestrator** (not directly to the stock agent), which then routes to the StockSpecializedAgent. This is a strict hub-and-spoke pattern — no spoke-to-spoke handoffs.
- [x] FR-3 — StockSpecializedAgent is **gated** behind `PROFILE_READY` — enforced in code by physically removing the stock agent from the handoff topology when `PROFILE_READY` is absent (not just prompt instructions). Users must complete their profile before accessing stock data.
- [x] FR-4 — The AdvisorAgent handles general non-stock financial advice (retirement, budgeting, bonds, insurance, tax planning) — behavior is more strictly scoped than before; stock-related queries are always redirected.
- [x] FR-5 — The StockSpecializedAgent returns the Foundry agent's response (including any document citations) to the user
- [x] FR-6 — The Orchestrator handles **unsupported specializations** directly (real estate, crypto, commodities, forex, options, futures) — informing the user and offering available alternatives without handing off to any agent

### User Scenarios

1. **Stock fundamentals query (profile not yet complete):**
   User: "What was Microsoft's net income in 2024?"
   → `PROFILE_READY` not found in history
   → Stock agent is **physically absent** from the handoff topology (code-enforced gating)
   → Orchestrator can ONLY route to ProfileAgent → must complete profile first
   → After profile completion, user re-asks → Stock agent is now in topology → Routes to StockSpecializedAgent

2. **Stock query after profile completion:**
   User has completed profile (`PROFILE_READY` exists in history)
   User: "How did Apple's revenue grow over the last 3 years?"
   → Workflow rebuilt with full agent access (advisor + stock agents added to topology)
   → Orchestrator detects stock-specific intent → Routes to `stock-specialized-investment-agent`
   → Returns grounded answer from Apple annual reports

3. **General non-stock advice (goes to AdvisorAgent):**
   User: "Help me plan for retirement"
   → Orchestrator detects general advice intent + `PROFILE_READY` exists → Routes to AdvisorAgent
   → AdvisorAgent provides personalized retirement planning guidance

4. **Ambiguous query — "what should I invest in?":**
   User: "What should I invest in?"
   → Orchestrator routes to `stock-specialized-investment-agent` (rule: when in doubt, if it touches stocks at all → stock agent)

5. **Advisor-to-stock escalation (via orchestrator, not direct):**
   User is in an advice session with AdvisorAgent.
   User: "What was Apple's actual revenue last year?"
   → AdvisorAgent detects stock-specific data request it cannot answer
   → Hands off to **orchestrator_agent** (not directly to stock agent)
   → Orchestrator routes to `stock-specialized-investment-agent`
   → StockSpecializedAgent answers with grounded data from Apple annual reports

6. **Unsupported specialization:**
   User: "What crypto should I buy?"
   → Orchestrator detects unsupported specialization → responds directly (no handoff):
   → "We don't currently offer specialized advisory for [area]. We can help you with stock investments or general financial advice."

### Out of Scope

- Creating or deploying the Azure Foundry agent (already pre-provisioned)
- Bing search grounding — the Foundry agent uses file search on uploaded stock documents only
- Combining stock data + personalized advice in a single response (future: AdvisorAgent could consult stock data)
- Adding new MCP tools — the existing `run_finwise_workflow` tool handles all routing internally
- New MCP server or CLI project
- Stock data APIs or real-time market data

### Open Questions (Functional)

- [x] Should stock queries be gated behind PROFILE_READY? → **Yes**, consistent with all other agents — enforced in code (not just prompts) by conditionally rebuilding the workflow topology
- [x] Should the orchestrator routing distinguish between "stock data" and "general financial advice" when PROFILE_READY exists? → **Yes** — stock-related queries (broad match, including "what should I invest in?") → StockAgent, general non-stock advice (retirement, budgeting, bonds) → AdvisorAgent, unsupported specializations (crypto, real estate) → Orchestrator responds directly

```
        │ Agent    │   │ Agent    │   │ Investment Agent   │ ← NEW
        │ (SPOKE)  │   │ (SPOKE)  │   │ (SPOKE)            │
```

**Key distinction**: The Profile and Advisor agents are `ChatClientAgent` instances created locally (local LLM + local prompt). The Stock Specialized Agent is a **Foundry-hosted `AIAgent`** resolved once at startup —  its LLM model, prompt/instructions, and file search grounding are all defined and managed in Azure AI Foundry, not in local code. **Handoff paths (strict hub-and-spoke — no spoke-to-spoke):**
- Orchestrator → ProfileAgent, AdvisorAgent, StockSpecializedAgent (hub-to-spoke)
- ProfileAgent, AdvisorAgent, StockSpecializedAgent → Orchestrator (spoke-to-hub)

**Code-enforced profile gating:**
```csharp

    ? [profileAgent, advisorAgent, stockAgent]
    : [profileAgent];  // Only profile agent until PROFILE_READY
```

Without `PROFILE_READY`, the advisor and stock agents **physically don't exist** as handoff targets. The orchestrator literally cannot route to them. No prompt compliance required.

**Safety guards:**
- 60-second `CancellationToken` timeout prevents runaway workflows
- 25-invocation max guard (`MaxAgentInvocations`) prevents infinite handoff loops
- Both produce user-friendly error messages instead of HTTP timeouts

### Implementation Approach: Native Foundry Agent via AIProjectClient SDK

The StockSpecializedAgent is the **pre-provisioned Azure Foundry native `AIAgent`**, resolved once at startup via `AIProjectClient.GetAIAgentAsync(agentName)` and placed directly into the workflow's handoff topology alongside the local `ChatClientAgent` spokes. There is no local wrapper, no intermediary, and no double-LLM call — the orchestrator hands off directly to the Foundry agent, which processes the query using its own LLM model, instructions, and file search grounding (all managed in Azure AI Foundry, not in local code).

**Why a native Foundry agent (not a tool-based wrapper)?**
- Zero overhead — the orchestrator hands off directly to the Foundry agent with no intermediary LLM call
- Everything that defines the agent (model, instructions, file search grounding) lives in Azure AI Foundry — single source of truth, no local prompt duplication
- The `Microsoft.Agents.AI.AzureAI` bridge makes the Foundry `AIAgent` a first-class peer in the `AgentWorkflow`, indistinguishable from local `ChatClientAgent` instances at the workflow level
- `ClientSecretCredential` (service principal auth) is available, solving the initial API key limitation that was observed with `AIProjectClient`

**Component layout (1 file — differs from other agents' 2-file pattern):**

```
src/FinWise.MultiAgentWorkflow/
  Agents/
    StockSpecializedAgent/
      StockSpecializedAgentFactory.cs     ← Factory: resolves Foundry AIAgent via AIProjectClient
    temp/
      StockSpecializedAgent.prompt.md     ← Reference prompt (not loaded by factory)
```

The factory does NOT create a `ChatClientAgent`. It receives `AIProjectClient` and `agentName`, and its `CreateAgentAsync()` method resolves the pre-provisioned Foundry agent. The returned `AIAgent` is the Foundry agent itself — the same object is placed directly into the `availableAgents` array in `CreateAgentsAndWorkflow()`. Everything that defines it (LLM model, system prompt, file search tool, grounded documents) lives in Azure AI Foundry, not in local code.

**Data flow:**

```
Orchestrator ──handoff_to_stock-specialized-investment-agent──▶ Foundry AIAgent
                                              │                  (stock-specialized-investment-agent)
                                              │
                                              │ The Foundry agent processes the query
                                              │ directly in Azure AI Foundry:
                                              │   • Uses its own LLM model
                                              │   • Uses its own system instructions
                                              │   • Runs file search on grounded stock docs
                                              │     (AAPL, MSFT, TSLA, NVDA, AMZN)
                                              ▼
                                        Response with citations
                                              │
                                              │ handoff_to_orchestrator_agent
                                              ▼
                                        Orchestrator returns to user
```

### Orchestrator Routing Updates

The `OrchestratorAgent.prompt.md` decision tree expands from 2-spoke to 3-spoke with **broad stock matching** and **unsupported specialization handling**:

```
STEP 1: Search entire history for PROFILE_READY marker

IF PROFILE_READY NOT FOUND:
  → ALWAYS route to profile_agent (no exceptions — code-enforced too)

IF PROFILE_READY FOUND:
  Check user intent:
  ├─ ANYTHING related to stocks (broad match) → stock-specialized-investment-agent
  │  (stocks, shares, equities, tickers, stock market, stock picks,
  │   company financials, annual reports, revenue, earnings, "what should
  │   I buy/invest in", specific company names like Apple/Microsoft/Tesla)
  │  Rule: when in doubt between advisor and stock agent,
  │        if the query touches stocks at all → stock agent
  ├─ Unsupported specialization (crypto, real estate, commodities,
  │  forex, options, futures) → Orchestrator responds DIRECTLY (no handoff)
  │  "We don't currently offer specialized advisory for [area]..."
  ├─ Profile-related intent (show, update, delete) → profile_agent (unchanged)
  └─ General non-stock advice (retirement, budgeting, bonds,
     insurance, tax planning) → advisor_agent
```

**Key changes from original spec**:
- Stock agent routing is **much broader** than originally planned — any query touching stocks goes there
- Generic "what should I invest in?" routes to stock agent (not advisor)
- Unsupported specializations handled directly by orchestrator (new behavior)
- Handoff function name uses hyphens: `handoff_to_stock-specialized-investment-agent` (matches Foundry agent's actual name)

### AdvisorAgent Handoff Behavior

The AdvisorAgent does **NOT** have a direct handoff to the StockSpecializedAgent. Instead, it hands off to the **orchestrator** for all escalations. This maintains the strict hub-and-spoke pattern.

```
AdvisorAgent decision:
  IF user asks SPECIALIZED investment question (stocks, real estate,
     crypto, commodities, "what should I buy?"):
    → handoff_to_orchestrator_agent (orchestrator routes appropriately)
  IF PROFILE_READY not found:
    → handoff_to_orchestrator_agent (silent, no text)
  ELSE:
    → Continue providing general non-stock financial advice
```

This differs from the original spec which proposed a spoke-to-spoke handoff. The hub-and-spoke pattern was chosen to keep routing logic centralized in the orchestrator and avoid complex multi-agent routing rules.

### Alternatives Considered

| Approach | Pros | Cons | Decision |
|----------|------|------|----------|
| **Native Foundry AIAgent via AIProjectClient SDK** (chosen) | No double-LLM call, Foundry agent participates directly in handoff workflow, leverages `Microsoft.Agents.AI.AzureAI` bridge | Adds NuGet dependencies (`Azure.AI.Projects`, `Azure.Identity`), factory doesn't follow 2-file pattern of other agents | ✅ Implemented — cleanest integration, best performance |

## Learnings & Notes

> Insights discovered during implementation. See also [journal/03- The Ghost in the Session.md](../../journal/03-%20The%20Ghost%20in%20the%20Session.md) for the narrative account.

### Patterns Discovered

- `AIProjectClient.GetAIAgentAsync(agentName)` resolves a Foundry-hosted agent as an `AIAgent` object that can be placed directly into the local `AgentWorkflow` handoff topology — the Foundry agent becomes a first-class peer alongside local `ChatClientAgent` instances, with no wrapper or intermediary. The `Microsoft.Agents.AI.AzureAI` bridge package makes this possible.
- The `AgentWorkflowBuilder.WithHandoffs()` API makes adding new spokes straightforward — just add to the orchestrator's handoff list
- **Spoke-to-spoke handoffs were rejected in favor of strict hub-and-spoke** — keeping routing logic centralized in the orchestrator proved simpler and more maintainable
- **Code-enforced gating > prompt-only gating**: LLM instructions are probabilistic; physically removing agents from the topology makes invalid routing impossible
- **Safety guards are mandatory for multi-agent workflows**: Any bidirectional handoff topology can loop. `CancellationToken` timeouts and max-invocation guards are safety requirements, not optimizations.

### Research: Authentication Decision

**Original finding**: The `Azure.AI.Projects` SDK (`AIProjectClient`) does NOT support API key authentication — only `TokenCredential`.

**Resolution**: Instead of bypassing the SDK with direct `HttpClient` calls (as originally proposed), the implementation uses `ClientSecretCredential` (service principal authentication via Azure.Identity). This:
- Keeps the `AIProjectClient` SDK for proper agent resolution and management
- Enables the `Microsoft.Agents.AI.AzureAI` bridge for native workflow integration
- Follows Azure best practices for service-to-service authentication
- Requires 3 environment variables (tenant ID, client ID, client secret) instead of 1 API key

The original spec's decision to bypass the SDK was based on the constraint "API key auth required" — but switching to service principal auth eliminated that constraint entirely.

### Issues Encountered

1. **SDK RC4 broke InMemoryChatHistoryProvider**: The `GetService<InMemoryChatHistoryProvider>()` API broke silently during the preview → RC4 upgrade. Messages were serialized but not restored. Fix: independent message serialization alongside the AgentSession blob.

2. **Infinite handoff loop**: Without code-enforced gating, `gpt-4o-mini` would route to `advisor_agent` even without `PROFILE_READY`, causing advisor ↔ orchestrator infinite bouncing (198 invocations in 100 seconds). Fix: conditionally build the workflow topology based on `isProfileReady`.

3. **Orchestrator routing too narrow**: Initial routing table only sent "company financials, annual reports, revenue" to the stock agent. But the Foundry agent handles everything stock-related (picks, recommendations, analysis). Fix: broadened routing — any stock-touching query → stock agent.
