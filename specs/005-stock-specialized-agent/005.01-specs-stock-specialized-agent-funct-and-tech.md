## Functional Specification

### Problem Statement

The FinWise multi-agent workflow currently has two agents: **UserProfileAgent** (profile collection) and **AdvisorAgent** (personalized investment advice). The AdvisorAgent provides general investment guidance but has no access to real company financial data — it cannot answer questions about specific stock fundamentals, annual report details, or company financials.

An Azure Foundry agent (`stock-specialized-investment-agent`) already exists and is **pre-provisioned** in Azure AI Foundry, grounded on actual stock documents (Apple, Microsoft, Tesla, Nvidia, and Amazon annual reports). This agent needs to be integrated into the existing FinWise multi-agent workflow as a new spoke agent so users can ask stock-specific questions and get answers backed by real financial documents.

### Proposed Solution

Add a **StockSpecializedAgent** as a fourth agent (third spoke) in the existing hub-and-spoke multi-agent workflow. Unlike the other agents which are `ChatClientAgent` instances created locally with a local prompt and the Azure OpenAI `IChatClient`, this agent is the pre-provisioned **Azure Foundry native `AIAgent`** — its LLM model, prompt/instructions, and file search grounding are all defined and managed in Azure AI Foundry, not in local code. It is resolved once at startup via `AIProjectClient.GetAIAgentAsync()` and placed directly into the `AgentWorkflow` handoff topology as a peer of the other spokes. The `Microsoft.Agents.AI.AzureAI` bridge package enables this Foundry agent to participate seamlessly in the Microsoft Agent Framework's handoff mechanism.

**User interaction flow:**
1. User completes profile (existing flow, unchanged)
2. User asks a stock-specific question (e.g., "What was Apple's revenue in 2024?")
3. Orchestrator hands off directly to the Foundry-hosted stock agent (no local intermediary)
4. The Foundry agent searches its grounded stock documents and returns an answer
5. Response flows back through the workflow to the user

**No new MCP server. No new CLI.** The agent is added directly into `FinWise.MultiAgentWorkflow` as a new spoke in the existing handoff topology.

### Functional Requirements

- [x] FR-1 — Users can ask stock-specific questions (company financials, annual reports, revenue, earnings) and receive answers grounded in real financial documents
- [x] FR-2 — The Orchestrator routes stock/company-specific queries to the StockSpecializedAgent and personalized advice queries to the AdvisorAgent. Routing is broad: anything touching stocks at all goes to the stock agent.
- [x] FR-2b — The AdvisorAgent hands off stock-specific questions to the **orchestrator** (not directly to the stock agent), which then routes to the StockSpecializedAgent. This is a strict hub-and-spoke pattern — no spoke-to-spoke handoffs.
- [x] FR-3 — StockSpecializedAgent is **gated** behind `PROFILE_READY` — enforced in code by physically removing the stock agent from the handoff topology when `PROFILE_READY` is absent (not just prompt instructions). Users must complete their profile before accessing stock data.
- [x] FR-4 — The AdvisorAgent handles general non-stock financial advice (retirement, budgeting, bonds, insurance, tax planning) — behavior is more strictly scoped than before; stock-related queries are always redirected.
- [x] FR-5 — The StockSpecializedAgent returns the Foundry agent's response (including any document citations) to the user
- [x] FR-6 — The Orchestrator handles **unsupported specializations** directly (real estate, crypto, commodities, forex, options, futures) — informing the user and offering available alternatives without handing off to any agent

### User Scenarios

1. **Stock query — profile not yet complete:**
   "What was Microsoft's net income in 2024?"
   → Stock agent is physically absent from topology (code-enforced gating) → Orchestrator can only route to ProfileAgent → must complete profile first

2. **Stock query — profile complete:**
   "How did Apple's revenue grow over the last 3 years?"
   → Workflow rebuilt with full agent access → Orchestrator routes to `stock-specialized-investment-agent` → Grounded answer from Apple annual reports

3. **General non-stock advice:**
   "Help me plan for retirement"
   → Routes to AdvisorAgent (retirement, budgeting, bonds — only when not stock-related)

4. **Ambiguous "what should I invest in?":**
   → Routes to `stock-specialized-investment-agent` (rule: when in doubt, if it touches stocks at all → stock agent)

5. **Advisor-to-stock escalation (via orchestrator):**
   User is with AdvisorAgent, asks "What was Apple's actual revenue last year?"
   → AdvisorAgent hands off to orchestrator → orchestrator routes to stock agent

6. **Unsupported specialization:**
   "What crypto should I buy?"
   → Orchestrator responds directly: "We don't currently offer specialized advisory for [area]."

### Out of Scope

- Creating or deploying the Azure Foundry agent (already pre-provisioned)
- Bing search grounding — the Foundry agent uses file search on uploaded stock documents only
- Combining stock data + personalized advice in a single response
- Adding new MCP tools — the existing `run_finwise_workflow` tool handles all routing internally
- Stock data APIs or real-time market data

## Technical Specification

### Architecture

**Four-agent hub-and-spoke topology:**

```
                    ┌────────────────────────┐
                    │  OrchestratorAgent     │
                    │  (HUB — silent         │
                    │   router, tool calls)  │
                    │                        │
                    │  Type: ChatClientAgent │
                    │  (local, Azure AOAI)   │
                    └──┬──────┬──────┬───────┘
                       │      │      │
              ┌────────┘      │      └────────┐
              ▼               ▼               ▼
        ┌──────────┐   ┌──────────┐   ┌────────────────────┐
        │ Profile  │   │ Advisor  │   │ Stock Specialized  │
        │ Agent    │   │ Agent    │   │ Investment Agent   │
        │ (SPOKE)  │   │ (SPOKE)  │   │ (SPOKE)            │
        │          │   │          │   │                    │
        │Type:     │   │Type:     │   │ Type:              │
        │ChatClient│   │ChatClient│   │ Foundry AIAgent    │
        │ Agent    │   │ Agent    │   │ (lives in Azure    │
        │ (local)  │   │ (local)  │   │  AI Foundry,       │
        └──────────┘   └──────────┘   │  grounded on:      │
                                      │  AAPL, MSFT, TSLA, │
                                      │  NVDA, AMZN docs)  │
                                      │                    │
                                      │ Resolved at startup│
                                      │ via AIProjectClient│
                                      │ .GetAIAgentAsync() │
                                      └────────────────────┘
```

**Key distinction**: Profile and Advisor are `ChatClientAgent` instances created locally (prompt + Azure OpenAI `IChatClient`). The Stock agent is a **Foundry-hosted `AIAgent`** — its model, instructions, and file search grounding live in Azure AI Foundry. In the workflow, it's the same `AIAgent` object placed directly into the handoff topology — no local wrapper or intermediary.

**Handoff paths** — strict hub-and-spoke: Orchestrator ↔ each spoke. No spoke-to-spoke handoffs.

**Code-enforced profile gating** — Without `PROFILE_READY` in message history, advisor and stock agents are physically absent from the `AgentWorkflow` topology. The orchestrator cannot route to them regardless of prompt instructions.

**Safety guards** — 60-second timeout and 25-invocation max prevent infinite handoff loops.

### Orchestrator Routing

The `OrchestratorAgent.prompt.md` decision tree:

```
STEP 1: Search entire history for PROFILE_READY marker

IF PROFILE_READY NOT FOUND:
  → ALWAYS route to profile_agent (code-enforced too — other agents absent)

IF PROFILE_READY FOUND:
  ├─ ANYTHING stock-related (broad match) → stock-specialized-investment-agent
  │  Includes: stocks, company financials, annual reports, revenue, earnings,
  │  stock picks, "what should I buy/invest in", company names (Apple, Tesla…)
  │  Rule: when in doubt, if it touches stocks at all → stock agent
  ├─ Unsupported specialization (crypto, real estate, commodities, forex,
  │  options, futures) → Orchestrator responds DIRECTLY (no handoff)
  ├─ Profile-related (show, update, delete) → profile_agent
  └─ General non-stock advice (retirement, budgeting, bonds,
     insurance, tax planning) → advisor_agent
```

The AdvisorAgent hands off **all** specialized questions (stocks, real estate, crypto, "what should I buy?") to the orchestrator — never directly to other spokes. Routing logic stays centralized.

### Alternatives Considered

| Approach | Decision | Rationale |
|----------|----------|-----------|
| **Native Foundry AIAgent via AIProjectClient** | ✅ Chosen | Zero overhead — orchestrator hands off directly. `Microsoft.Agents.AI.AzureAI` bridge makes it a first-class workflow peer. |
| **Tool-based wrapper + direct HTTP** (originally proposed) | ❌ Superseded | Extra LLM call per query, reimplements what SDK provides. |
| **Tool-based wrapper + Azure.AI.Projects SDK** | ❌ Rejected | Extra LLM call per query. Native agent eliminates wrapper entirely. |
| **Custom IChatClient adapter** | Deferred | Complex; `IChatClient` is chat-oriented while Foundry uses Responses API. |

## Implementation Summary

> All steps completed March 2026. See [journal/03- The Ghost in the Session.md](../../journal/03-%20The%20Ghost%20in%20the%20Session.md) for the full story.

**What was built:**
- `StockSpecializedAgentFactory` — Single factory file. Receives `AIProjectClient` + agent name, resolves Foundry agent via `GetAIAgentAsync()`. No local prompt (Foundry agent has built-in instructions). Reference prompt in `Agents/temp/`.
- Workflow integration — `FinWiseWorkflowService` takes `AIAgent stockAgent`; conditionally includes agents based on `PROFILE_READY` (code-enforced gating); 60s timeout + 25-invocation max guard.
- Orchestrator & Advisor prompt updates — broad stock matching, unsupported specialization handling, strict hub-and-spoke handoff rules.
- Composition root (`Program.cs`) — reads 5 env vars → `ClientSecretCredential` → `AIProjectClient` → resolves `AIAgent` at startup → injects into workflow.
- Session management fixes (unplanned — SDK RC4 migration) — independent message serialization after `InMemoryChatHistoryProvider` broke silently.

**Testing:**
- Unit tests (`FinWise.MultiAgentWorkflow.UnitTests`) — factory constructor validation
- Integration tests (`FinWise.StockAgent.IntegrationTests`) — agent resolution, Foundry connectivity, stock query response
- E2E tests (`FinWise.McpServer.IntegrationTests`) — stock routing through full MCP pipeline

## Learnings & Notes

> See also [journal/03- The Ghost in the Session.md](../../journal/03-%20The%20Ghost%20in%20the%20Session.md) for the narrative account.

### Key Patterns

- `AIProjectClient.GetAIAgentAsync()` resolves a Foundry-hosted agent as an `AIAgent` that can be placed directly into the local `AgentWorkflow` topology — first-class peer alongside `ChatClientAgent` instances, via the `Microsoft.Agents.AI.AzureAI` bridge.
- **Code-enforced gating > prompt-only gating** — LLM instructions are probabilistic; physically removing agents from the topology makes invalid routing impossible.
- **Safety guards are mandatory** — Any multi-agent workflow with bidirectional handoffs can loop. Timeouts and max-invocation guards are safety requirements.
- **Strict hub-and-spoke > spoke-to-spoke** — Centralizing routing in the orchestrator was simpler and more maintainable.

### Authentication Decision

The original spec proposed bypassing the SDK with direct `HttpClient` + `api-key` header (since `AIProjectClient` only supports `TokenCredential`). The implementation uses `ClientSecretCredential` (service principal) instead, keeping the SDK and enabling the `Microsoft.Agents.AI.AzureAI` bridge.

### Issues Encountered

1. **SDK RC4 silent breaking change** — `GetService<InMemoryChatHistoryProvider>()` stopped working on deserialized sessions. Fix: independent message serialization alongside the `AgentSession` blob.
2. **Infinite handoff loop** — Without code-enforced gating, the LLM would route to `advisor_agent` without `PROFILE_READY`, causing 198 invocations in 100 seconds. Fix: conditionally build the workflow topology.
3. **Routing too narrow** — Initial routing only sent "company financials, annual reports" to the stock agent. The Foundry agent handles everything stock-related. Fix: broad matching.

### Future Work

- **v0.3 session management** — Adopt SDK's `AgentSessionStore` abstract class and `StateBag`-based messages
- **Stable agent IDs** — Set `Id = "orchestrator_agent"` in `ChatClientAgentOptions` — prerequisite for SDK session store
- **Cross-agent consultation** — AdvisorAgent could consult StockSpecializedAgent for data-backed recommendations
- **Additional companies** — Foundry agent's file search can be expanded without code changes
